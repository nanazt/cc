**File structure** (per D-01: schema + guide approach):

1. **H1 heading:** `# Domain Service Template`
2. **Brief preamble** (2-3 lines): State this template defines the section schema for domain-service archetype. The spec-consolidator reads this to determine which sections to produce.
3. **Separator:** `---` followed by `# context.md` heading to delineate the context.md section schema.
4. **8 H2 sections for context.md** in this exact order (per IMPL-SPEC lines 127-135):

   - `## Domain Model` — Guide: entities, their fields, and constraints that define the service's core data structures. Table schema: Entity, Field, Type, Constraints.
   - `## Adapter Contracts` — Guide: trait interfaces and method signatures that define how the service communicates with external dependencies. NOT "Ports" (v1 name per D-13). Describe as the boundary between domain logic and infrastructure implementations.
   - `## Business Rules` — Guide: service-level rules that govern behavior across operations. Each rule has SR-{N} identifier and `(Source: Phase {id})` provenance. Table schema: ID, Rule, Source.
   - `## Error Categories` — Guide: error types the service can produce. Table schema: Category, Identifier, Description (per IMPL-SPEC line 131).
   - `## Service Interface` — Guide: operations this service exposes, using `{Service}.{Operation}` naming format. NOT "gRPC Interface" (v1 name per D-13). For each operation: inputs, outputs, key constraints.
   - `## Configuration` — Guide: environment variables and tunable parameters the service requires. Table schema: Variable, Default, Description.
   - `## Cross-Service Dependencies` — Guide: what this service requires from other services. Table schema: Service, Operation, Purpose.
   - `## State Management (conditional)` — Per D-05/D-06: heading includes `(conditional)` marker. First line: `**Include when:** this service manages long-lived stateful entities whose lifecycle spans multiple operations.` Guide: state transitions, lifecycle events, consistency guarantees.

5. **Separator:** `---` followed by `# cases.md` heading to delineate the cases.md format section.
6. **cases.md format** (per D-07, IMPL-SPEC lines 137-139): One example operation showing the full structure with placeholder notation:

   ```
   ## {Service}.{Operation}

   ### Service Rules

   | ID | Rule |
   |----|------|
   | SR-{N} | {rule description} `(Source: Phase {id})` |

   ### Operation Rules

   | ID | Rule |
   |----|------|
   | OR-{N} | {rule description} |

   ### Side Effects

   - {side effect description}

   ### Cases

   | Type | ID | Case | Expected Outcome |
   |------|----|------|------------------|
   | S | S1 | {success scenario} | {expected result including side effects} |
   | F | F1 | {failure scenario} | {expected error, side effects NOT triggered} |
   | E | E1 | {edge case scenario} | {expected behavior} |
   ```

**Content rules:**
- All guide text in technology-neutral language per D-09 (no "gRPC", "REST", "SQL", "Kafka", etc.)
- Only `{placeholder}` notation per D-01/D-02 — no example data, no real service names
- Each section: H2 heading, 1-2 sentence purpose description, table column schema where applicable
- Conditional section has `**Include when:**` line with behavioral condition per D-05/D-06

**File 1: `skills/consolidate/templates/gateway-bff.md`**

Structure:
1. **H1:** `# Gateway / BFF Template`
2. **Preamble** (2-3 lines): section schema for gateway-bff archetype.
3. **`# context.md`** section with 7 H2 sections in this exact order (per IMPL-SPEC lines 143-150):

   - `## Route/Endpoint Table` — Guide: routes and endpoints the gateway exposes. Table schema: Tier, Method, Path, Description, Source (per IMPL-SPEC line 144). NOT "Route Table" (v1 name).
   - `## Middleware Stack` — Guide: ordered middleware components and their purpose. Describe as the processing pipeline applied to requests. Table schema: Order, Name, Purpose.
   - `## Composition Patterns` — Guide: how the gateway composes calls to backend services for each user-facing operation. NOT "Orchestration Patterns" (v1 name per D-13). Describe strategies for combining multiple service responses.
   - `## Error Translation` — Guide: how errors from backend services are translated for external consumers. NOT "Error Handling" (v1 name per D-13). Include the mapping format. Table schema: Internal Category, External Code, External Message.
   - `## External API Conventions` — Guide: naming conventions, pagination patterns, and request/response format rules for the external interface. NOT "REST Conventions" (v1 name per D-13).
   - `## Configuration` — Guide: environment variables, listen address, and timeout settings. Table schema: Variable, Default, Description.
   - `## Identity Propagation (conditional)` — Per D-05/D-06: heading includes `(conditional)` marker. First line: `**Include when:** caller identity must flow through the gateway to backend services for authorization or audit purposes.` Guide: how identity tokens/context are extracted, validated, and forwarded.

4. **`# cases.md (conditional)`** section (per D-12, IMPL-SPEC line 152):
   - **Condition guide at the top** (per D-12): Explain the "judgment seat" test in technology-neutral behavioral terms:
     - **Qualifies:** Operations where the code examines state, applies rules, and branches to different outcomes. Operations with compensation logic across multiple service calls. Middleware that makes conditional decisions based on request properties.
     - **Does NOT qualify:** Translation or format conversion between external and internal representations. Passthrough routing to a single backend service. Deterministic mapping with no conditional branching.
   - Below the condition guide, same cases.md format as domain-service.md: one `## {Service}.{Operation}` example with SR-N, OR-N, Side Effects, Cases table using `{placeholder}` notation.

First line: `**Include when:** this service performs stateful event processing where the outcome depends on accumulated state from prior events.` Guide: state transitions, consistency guarantees for event-sourced or saga patterns.
