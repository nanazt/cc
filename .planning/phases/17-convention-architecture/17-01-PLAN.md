**1. Source Structure (ARCH-01)**
Conventions live in `conventions/` at project root. Each convention area is a subdirectory with short, intuitive kebab-case name. Files within:
- `CONVENTION.md` -- base convention (optional, only when it passes delta test)
- `{lang}.md` -- language-specific convention (e.g., `rust.md`, `typescript.md`, `python.md`)
- `hooks/` -- enforcement scripts (optional)

Include the canonical source structure example from D-06:
```
conventions/
+-- commit/
|   +-- CONVENTION.md
+-- coding/
|   +-- CONVENTION.md
|   +-- rust.md
+-- test/
|   +-- CONVENTION.md
|   +-- rust.md
+-- error/
    +-- rust.md
```

**2. Convention File Format (D-01, D-11, D-12, D-13)**
Conventions are Claude Code Rules (installed to `.claude/rules/`). Frontmatter uses ONLY Claude Code standard fields:
- `description` -- human-readable summary (always present)
- `paths` -- file glob for path-scoped loading (optional; omitted = always loads)
- `alwaysApply` -- `false` when `paths` is present (path-scoped), omitted or `true` for always-loading

Follow the official Claude Code documentation format. Do NOT document upstream parser bugs or workarounds -- that is Claude Code's responsibility to fix.

Format for path-scoped (language-specific) conventions -- show as frontmatter fields:
  description: "Rust coding conventions"
  alwaysApply: false
  paths:
    - "**/*.rs"

Format for always-loading (base) conventions -- show as frontmatter fields:
  description: "Commit message format and quality standards"
  (no paths field, no alwaysApply field -- or alwaysApply: true)

**3. Base and Language-Specific Relationship (D-07, D-08, D-09, D-10, ARCH-04)**
- Base is OPTIONAL. Only created when it passes the delta test (ARCH-03)
- When base exists: language-specific conventions are ADDITIVE (extend, never duplicate base content). Claude loads both files
- When base does not exist: language-specific conventions are STANDALONE (fully self-contained)
- Within same area: base + language-specific installed together
- Across areas: conventions are INDEPENDENT. No cross-references. Each installable alone

**4. Delta Test (ARCH-03, D-15, D-23)**
Definition: Each rule in a convention must pass: "Would Claude already do this without being told?"
Three criteria:
1. Default behavior test -- does Claude already follow this rule?
2. Style divergence test -- does the user want behavior different from Claude's default?
3. Consistency test -- would Claude's behavior vary across sessions without this rule?

The delta test is embedded in the /convention skill's authoring process (Phase 18). No separate validation infrastructure. It naturally keeps conventions concise (no explicit size limit needed).

A convention that fails the delta test entirely (all rules are things Claude already knows) should not be created. This is why base conventions are optional.

**5. Naming and Installation (D-19, D-20, D-21)**
Source-to-installed mapping:
| Source | Installed As | Rule |
|--------|-------------|------|
| `conventions/commit/CONVENTION.md` | `.claude/rules/cckit-commit.md` | `cckit-{area}.md` |
| `conventions/coding/CONVENTION.md` | `.claude/rules/cckit-coding.md` | `cckit-{area}.md` |
| `conventions/coding/rust.md` | `.claude/rules/cckit-coding-rust.md` | `cckit-{area}-{lang}.md` |
| `conventions/error/rust.md` | `.claude/rules/cckit-error-rust.md` | `cckit-{area}-{lang}.md` |

Installer discovers conventions by scanning `conventions/` directory. No manifest file. Presence of `CONVENTION.md` or `{lang}.md` = convention exists.

Self-application: `deno run tools/install.ts --self`

**6. Hook Architecture (D-16, D-17)**
Hooks live inside the convention directory they enforce:
```
conventions/commit/
+-- CONVENTION.md
+-- hooks/
    +-- validate.sh
```

Hooks are shell scripts for Claude Code's PreToolUse/PostToolUse system. They receive JSON on stdin and output JSON with `permissionDecision` (allow/deny) and `permissionDecisionReason`.

Installed as: `conventions/commit/hooks/validate.sh` -> `.claude/hooks/cckit-commit-validate.sh`

Hook registration: installer must add hook configuration to host project's `.claude/settings.json`. Document the settings.json structure needed:
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [".claude/hooks/cckit-commit-validate.sh"]
      }
    ]
  }
}
```

Reference: existing `.claude/hooks/validate-commit-scope.sh` in cckit.

**7. Convention Content Rules (D-14)**
No prescribed internal structure. The /convention skill (Phase 18) determines optimal structure per convention area. Convention content is directives, not explanations -- every line should be actionable ("Always X", "Never Y", "When Z, do W").

**8. Override and Precedence (D-22)**
Host project's CLAUDE.md always takes precedence over cckit rules. This is Claude Code's built-in priority -- no custom override mechanism needed. Users can customize installed convention files (including `paths` values per D-13).

**9. Anti-Patterns**
List the anti-patterns from research:
- Convention-as-skill (use rules, not skills)
- Custom frontmatter fields (only `description`, `paths`, `alwaysApply`)
- Manifest file (directory scan is the discovery mechanism)
- Prescribed content structure (each convention's structure is skill-determined)
- Monolithic convention file (one file per convention area + language)
- Cross-convention references (each convention must be installable alone)

**Document format notes:**
- Follow the style of docs/MODEL.md -- specification-grade, not blog-post
- Use tables for structured mappings
- Include the version, status, and applicability header similar to MODEL.md
- Keep the "tech pack" term out entirely -- use "language-specific convention" throughout
