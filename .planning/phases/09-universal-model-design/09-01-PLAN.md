**Section 5: ## Schema Format** (per D-14, D-15, D-16, D-17, D-18)
- State: each host project has one schema file at `.planning/consolidation.schema.md`
- State: schema file is Markdown with tables (human-readable, machine-parseable)
- Subsection `### Schema Structure` -- define the complete format:

```markdown
# Consolidation Schema

A component is the smallest independently specifiable unit in your project.

## Meta

| Key | Value |
|-----|-------|
| version | 1 |
| rule-prefix | CR |
| e2e-flows | false |

## Components

| Component | Description |
|-----------|-------------|
| {name} | {what it does} |

## Sections: default

### Context Sections
1. **Overview** -- What this component does and why it exists
2. **Public Interface** -- Operations, commands, endpoints, or API surface this component exposes to consumers
3. **Domain Model** -- Entities, types, and data structures this component owns
4. **Behavior Rules** -- Business rules, constraints, and invariants governing this component's behavior
5. **Error Handling** -- Error categories, failure modes, and recovery strategies
6. **Dependencies** -- What this component requires from other components or external systems
7. **Configuration** -- Environment variables, feature flags, and tunable parameters

### Conditional Sections
- **State Lifecycle** -- Include when: component manages stateful entities with lifecycle transitions
- **Event Contracts** -- Include when: component produces or consumes events/messages
```

- Subsection `### Meta Fields` -- table with 4 columns: Key, Type, Default, Description
  - version | integer | 1 | Schema format version. Future versions may add new fields.
  - rule-prefix | string | CR | Prefix for component-level rules. Default "CR" (Component Rule).
  - e2e-flows | boolean | false | When true, enables cross-component E2E flow generation. Projects without cross-component communication should leave this false.
- State explicitly per D-18: no `specs-dir` or `operation-prefix` meta fields. `specs/` directory and `{Component}.{Operation}` format are fixed conventions.
- Subsection `### Parsing Rules` -- explicit parsing rules for Phase 10 implementers:
  - The `## Meta` heading identifies the meta section. The table below it contains key-value pairs.
  - The `## Components` heading identifies the component registry. Each row is one component. The table must have at least `Component` and `Description` columns.
  - The `## Sections: default` heading identifies the default section structure. The `### Context Sections` numbered list defines mandatory sections. The `### Conditional Sections` bullet list defines conditional sections.
  - Missing `## Meta` section: use defaults (version=1, rule-prefix=CR, e2e-flows=false).
  - Missing `## Sections: default` section: use the built-in default (the 7+2 sections defined in this spec).
  - Empty `## Components` table: valid schema, but consolidation will prompt for component discovery.

**Section 6: ## Component Discovery** (per D-21, D-22, D-23)
- Subsection `### Discovery Algorithm` -- numbered steps:
  1. **Primary: CASES.md heading scan.** Extract component names from `## {Component}.{Operation}` headings in the phase CASES.md. String parsing -- split on `.`, take the first segment. High accuracy.
  2. **Fallback: CONTEXT.md scan.** When CASES.md is absent or empty, scan CONTEXT.md for component references matched against the schema Components table. Agent inference -- lower accuracy.
  3. **On miss:** If both stages yield zero components, prompt the developer: "No components detected. Which components does this phase affect?"
- Subsection `### Schema Authority` -- state per D-22:
  - The schema is the authoritative component registry.
  - Components discovered during consolidation that are not in the schema trigger: "Component '{name}' not in schema. Add it?" prompt.
  - Schema grows automatically (new components added during consolidation). Deletion and rename detection are deferred to Phase 11 (orchestrator implementation).
  - Schema changes are reviewed at the consolidation confirmation step.

**Section 7: ## Conditional Section Evaluation** (per D-24, D-25, D-26)
- State the mechanism: spec-consolidator agent evaluates natural language conditions from the schema against phase documents (CONTEXT.md, CASES.md).
- For each conditional section, provide evaluation guidance:
  - **State Lifecycle** -- condition: "component manages stateful entities with lifecycle transitions"
    - Evidence for inclusion: phase documents mention state transitions (created/active/expired, draft/published/archived, pending/approved/rejected), entity status fields, lifecycle hooks, state machine references.
    - Evidence for exclusion: component is stateless, processes requests without maintaining entity state, no status/state fields mentioned.
    - Example include: "CONTEXT.md describes order states: placed -> paid -> shipped -> delivered"
    - Example exclude: "Component is a pure data transformer with no persistent state"
  - **Event Contracts** -- condition: "component produces or consumes events/messages"
    - Evidence for inclusion: phase documents mention event emission, message queues, pub/sub, webhooks, event handlers, listeners, callbacks triggered by external events.
    - Evidence for exclusion: component operates synchronously with no event production or consumption, uses only request/response patterns.
    - Example include: "CASES.md mentions 'emits UserCreated event after registration'"
    - Example exclude: "CLI command reads input, processes, writes output -- no events"
- Subsection `### Decision Logging` -- state per D-25:
  - Inclusion/exclusion reasoning is logged as HTML comments in the spec file, immediately before the section heading (if included) or at the end of context.md (if excluded).
  - Format for inclusion: `<!-- {Section Name}: Included -- {evidence summary} -->`
  - Format for exclusion: `<!-- {Section Name}: Excluded -- {evidence summary} -->`
  - Example: `<!-- State Lifecycle: Included -- CONTEXT.md mentions session state transitions: created -> active -> expired -->`
  - This serves dual purpose: traceability AND chain-of-thought accuracy improvement for the agent.
- State per D-26: future conditional sections are added by updating cckit's default schema definition. Users can also add or remove conditional sections in their own schema file.

**Section 8: ## Merge Rules** (per D-27)
Rewrite all 11 v1 merge rules in v2 terminology. Write from scratch using v2 terms -- do NOT copy-paste from IMPL-SPEC.md and find-replace. Include the v1-to-v2 terminology mapping table at the top of the section.

Terminology mapping table:
| Rule # | v1 Term | v2 Term |
|--------|---------|---------|
| 1 | `{Service}.{Op}` | `{Component}.{Op}` |
| 2 | PR to SR promotion | PR to CR promotion |
| 8 | "archetype sections" | "schema-defined sections" |

Then all 11 rules:
1. **Operation-level replacement:** When a later phase re-specifies an operation (same `{Component}.{Op}` heading), the entire operation section (rules, side effects, cases table) replaces the prior version. No case-level merge.
2. **PR to CR promotion:** All Phase Rules (PR-N) mechanically promote to Component Rules (CR-N). Renumber sequentially from highest existing CR + 1 to avoid collisions.
3. **TR exclusion:** Temp Rules (TR-N) are skipped entirely. They never enter specs/.
4. **R to OR transformation:** Operation Rules labeled `R-N` in source CASES.md are renamed to `OR-N` in consolidated output.
5. **GR reference only:** Global Rules (GR-XX) are referenced with `See GR-XX` notation. Never duplicated into specs/.
6. **Superseded operations:** For each entry in superseded operations table, remove the old operation section from the existing spec.
7. **Superseded rules:** For each entry in superseded rules table, skip the referenced PR during CR promotion.
8. **Section-level rewrite for context.md:** For each schema-defined section, if the new phase has content, rewrite the entire section with merged content (latest wins). If no new content, leave existing section unchanged.
9. **Provenance:** Every rule and significant decision includes `(Source: Phase {id})` or `(Source: Phase {id} D-{n})` inline provenance.
10. **Forward Concerns exclusion:** Forward Concerns from CASES.md are never consolidated into specs/.
11. **Exclusions from specs/:** Infrastructure setup, testing strategy, discussion rationale, research findings, planning artifacts.
