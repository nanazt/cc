- name: convention-researcher

- tools: Read, Grep, Glob, WebSearch, WebFetch (no Write -- researcher returns results, does not write files per D-14)

**Body structure (following case-briefer.md pattern):**

1. **Role description** -- one paragraph explaining the agent's purpose: research best practices, evaluate libraries, identify delta test candidates for a convention area. Emphasize: return structured report, do NOT interact with user, do NOT write convention files.

2. **Input Contract** -- XML tag table (following case-briefer pattern):
   - `<objective>` (required): mission statement with convention area and language
   - `<convention_area>` (required): the area name (e.g., "commit", "coding", "test")
   - `<target_language>` (required): language name or "none" for base-only research
   - `<host_project_context>` (required): summary of host project (CLAUDE.md content, technology stack, existing conventions) so research is project-aware
   - `<research_tools>` (optional): additional MCP tool names from cckit.json convention.tools config, merged with default tools
   - `<existing_convention>` (optional): content of existing convention file when researching for an update (so researcher can focus on gaps/improvements)

3. **Methodology** section with ordered steps:
   - Step 1: Parse inputs -- understand convention area, target language, host project context
   - Step 2: Research best practices -- use WebSearch and WebFetch to find authoritative sources (official style guides, widely-adopted community standards, language RFCs). Search for: "{area} best practices", "{area} conventions {language}", "{area} style guide". Follow links to primary sources, not just summaries.
   - Step 3: Evaluate libraries (language-specific only) -- when target_language is not "none", research available tooling/libraries. For each candidate library, evaluate: maintenance status (last release, commit frequency), community adoption (downloads, stars, usage in major projects), API stability (breaking changes history, semver adherence), documentation quality, security track record, ecosystem compatibility. Produce two tables per D-15: health/soundness table and feature comparison table. End with a recommendation and reasoning.
   - Step 4: Identify delta test candidates -- for each discovered best practice, assess: "Would Claude already follow this without being told?" Mark practices as HIGH-DELTA (Claude would NOT do this by default, must teach), LOW-DELTA (Claude might do this inconsistently, worth codifying for consistency), or NO-DELTA (Claude reliably does this already, remove). This helps the downstream generator apply the delta test.
   - Step 5: Compile report -- assemble findings into the output format.

4. **Research Quality Bar** -- the agent determines its own depth (per D-12, full autonomy), but must meet these minimums:
   - At least 3 authoritative sources consulted (not just search result snippets)
   - Library evaluation (if applicable) must cover at least the top 3 libraries by adoption
   - Delta test assessment must be provided for every proposed rule
   - If research finds insufficient information (per D-33), include a `## Insufficient Coverage` section listing areas where findings are thin, so the orchestrator can report to the user

5. **Output Contract** -- structured report format with these sections:
   - Header: "# Research Report: {area} Convention{language clause}" with metadata (timestamp, sources, area, language)
   - "## Best Practices": numbered list of practices, each with Practice name, description, Source (URL), Delta assessment (HIGH-DELTA / LOW-DELTA / NO-DELTA with reasoning)
   - "## Library Evaluation" (only when target_language is not "none"): subsections for "### Health and Soundness" (table with columns: Library, Maintenance, Adoption, API Stability, Docs, Security, Ecosystem), "### Feature Comparison" (table with feature rows), "### Recommendation" (which library and why, note user makes final selection)
   - "## Common Pitfalls": numbered list of pitfalls specific to this convention area
   - "## Delta Test Summary": table with Total practices, HIGH-DELTA, LOW-DELTA, NO-DELTA counts
   - "## Insufficient Coverage" (conditional): areas where research was thin

6. **Return Protocol** (following case-briefer pattern):
   Success: `## RESEARCH COMPLETE` followed by `Area: {area} | Language: {language} | Practices: {count} | Sources: {count}`
   Failure: `## RESEARCH FAILED` followed by `Reason: {what went wrong}`

7. **Quality Gate** checklist:
   - [ ] All best practices have source attribution
   - [ ] Delta assessment provided for every practice
   - [ ] Library evaluation includes health/soundness table (if language-specific)
   - [ ] Library evaluation includes feature comparison table (if language-specific)
   - [ ] Recommendation provided with reasoning (if language-specific)
   - [ ] No convention file generated (researcher does NOT generate conventions)
   - [ ] No user interaction attempted (researcher does NOT use AskUserQuestion)
   - [ ] Host project context incorporated into research (not generic advice)
   - [ ] Report is structured and parseable by the orchestrator

**Content authoring rules (from CLAUDE.md):** The agent definition uses structural placeholders, not domain-specific examples. The agent researches whatever convention area it receives -- it does not assume commit, coding, or any specific area.

Create `agents/convention-generator.md` with the following structure:

**Frontmatter** (use standard agent frontmatter with YAML delimiters):
- name: convention-generator
- description: (multiline >) Generates convention files following the cckit convention architecture. Applies delta test to self-filter rules. Handles both creation and update (surgical edit) modes. Returns generated content with generation report.
- tools: Read, Grep, Glob, Write (no WebSearch, no WebFetch -- generator does not research)
- model: sonnet

**Body structure:**

1. **Role description** -- one paragraph: generate convention files (Claude Code rules) that follow docs/CONVENTIONS.md architecture. Apply delta test during generation to self-filter rules. Handle both creation mode (new convention) and update/surgical-edit mode (modify existing). Produce a generation report alongside the file. Do NOT interact with users. Do NOT perform research.

2. **Input Contract** -- XML tag table:
   - `<objective>` (required): mission statement specifying base or language-specific, area, language
   - `<convention_spec>` (required): instruction to read docs/CONVENTIONS.md for format specification
   - `<research_results>` (required): structured research report from the researcher agent (full content, not a reference)
   - `<user_preferences>` (required): collected preferences from orchestrator's interactive session
   - `<delta_test_instructions>` (required): embedded delta test criteria (always provided by orchestrator)
   - `<tech_neutrality_check>` (conditional): only present for base conventions, contains the neutrality test question
   - `<output_path>` (required): resolved file path where the convention should be written
   - `<mode>` (required): "create" or "surgical-edit"
   - `<existing_content>` (conditional, surgical-edit only): current convention file content
   - `<requested_changes>` (conditional, surgical-edit only): what to modify/add/remove
   - `<feedback>` (optional): user feedback from preview rejection, for re-generation

3. **Generation Process** section (ordered steps):

   **Create mode:**
   - Step 1: Read docs/CONVENTIONS.md to understand the authoritative file format (frontmatter fields, paths syntax, structure rules)
   - Step 2: Parse research results -- extract HIGH-DELTA and LOW-DELTA practices (skip NO-DELTA)
   - Step 3: Parse user preferences -- these override research recommendations where they conflict
   - Step 4: Apply delta test per rule -- for each candidate rule from research + preferences:
     - "Would Claude already follow this rule without being told?" -> YES -> REMOVE, record in report
     - "Does the user want behavior different from Claude's default?" -> YES -> KEEP regardless of above
     - "Would Claude's behavior vary across sessions without this rule?" -> YES -> KEEP for consistency
     Record every removal decision with reasoning in the generation report.
   - Step 5: Apply tech-neutrality check (base conventions only, per D-21) -- for each remaining rule, test: "Could a Rust CLI, Go API, Python ML pipeline, and documentation site all follow this rule?" If no -> move to generation report as "tech-specific, belongs in language pack"
   - Step 6: Organize surviving rules into the convention file. Internal structure is at generator's discretion (per Phase 17 architecture: "No prescribed internal structure"). Use headings and grouping that best serves the convention area.
   - Step 7: Write frontmatter following docs/CONVENTIONS.md exact format:
     - Base convention: `description` field only (no paths, no alwaysApply)
     - Language-specific: `description`, `paths: {glob}` (unquoted), `alwaysApply: false`
     - CRITICAL: paths must be a single unquoted glob, not YAML array, not quoted string (Claude Code issue #17204)
   - Step 8: Add footer comment: `<!-- Generated by /convention, {ISO 8601 timestamp with seconds} -->` (per D-34)
   - Step 9: Write file to `<output_path>` using Write tool
   - Step 10: Compile generation report

   **Surgical-edit mode:**
   - Step 1: Read existing convention content from `<existing_content>`
   - Step 2: Parse `<requested_changes>` -- categorize as: rules to modify, rules to add, rules to remove
   - Step 3: For new/modified rules, apply delta test (same as create mode Step 4)
   - Step 4: Apply changes to existing content, preserving structure and unchanged rules
   - Step 5: Update footer timestamp
   - Step 6: Write updated file to `<output_path>`
   - Step 7: Compile change report (rules added, removed, modified with reasoning)

4. **Convention File Quality Rules:**
   - Content is directives, not explanations (per docs/CONVENTIONS.md: "Convention files are instructions to Claude, not documentation for humans")
   - Every line should be actionable -- no rationale paragraphs, no history
   - No cross-convention references (each convention is independently installable)
   - Zero cckit-specific frontmatter fields (only description, paths, alwaysApply)
   - No hardcoded project references (technology neutral for base, language-specific for lang files)

5. **Output Contract:**

   The generator writes the convention file AND returns a generation report in its return message. The report format:

   Success header: `## GENERATION COMPLETE`
   Then metadata: Mode (create/surgical-edit), Output (path), Convention type (base/language-specific)
   Then sections:
   - `### Rules Included`: Total count + numbered list of rule summaries
   - `### Rules Removed by Delta Test`: Total count + numbered list with removal reasoning per rule
   - `### Tech-Neutrality Results` (base only): Passed count, Failed count with list of failed rules
   - `### Changes Made` (surgical-edit only): Added/Modified/Removed counts with lists
   - `### Confidence`: HIGH/MEDIUM/LOW with reasoning

   Failure header: `## GENERATION FAILED` followed by `Reason: {what went wrong}`

6. **Edge Cases:**
   - All rules filtered by delta test (empty convention): return `GENERATION COMPLETE` with `Rules Included: Total: 0` and a note: "All candidate rules were filtered by the delta test. The orchestrator should report this to the user per the empty-convention protocol."
   - User feedback re-generation: when `<feedback>` is present, treat it as additional constraints on top of existing preferences. Do NOT start from scratch -- adjust the previous generation.
   - Missing research results: return `GENERATION FAILED` with reason "No research results provided"

7. **Quality Gate:**
   - [ ] Convention file follows docs/CONVENTIONS.md format exactly
   - [ ] Frontmatter uses only standard Claude Code fields (description, paths, alwaysApply)
   - [ ] paths field (if present) is unquoted glob with alwaysApply: false
   - [ ] Every rule passed the delta test (or was kept due to user preference override)
   - [ ] Base conventions contain zero language-specific or framework-specific rules
   - [ ] Footer comment present with ISO 8601 timestamp
   - [ ] Generation report included in return message
   - [ ] No user interaction attempted
   - [ ] No research performed (generator does not search the web)
   - [ ] Content is directives, not explanations
