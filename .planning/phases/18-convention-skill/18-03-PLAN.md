Title: `# Step: Research`

**Step 1: Build Host Project Context**

Gather host project context to pass to the researcher (per D-13):
- Read the host project's CLAUDE.md (if it exists) for technology stack, conventions, constraints
- Scan for package.json, Cargo.toml, go.mod, pyproject.toml, etc. to identify technology stack
- Check for existing convention files in .claude/rules/ or conventions/ to understand what's already installed
- Summarize into a compact context block (not the full files -- a summary paragraph)

**Step 2: Dispatch Research Agent**

Display progress announcement (per D-07): "Researching {area} conventions{lang clause}..."

Build the dispatch prompt for convention-researcher:

```
Agent(
  subagent_type: "convention-researcher",
  prompt: "<objective>
Research best practices, community conventions, and library options
for the '{area}' convention area{' targeting ' + lang if lang else ''}.
</objective>

<convention_area>{area}</convention_area>
<target_language>{lang or 'none'}</target_language>

<host_project_context>
{compiled host project context from Step 1}
</host_project_context>

<research_tools>
Default: WebSearch, WebFetch, Read, Grep, Glob
Additional: {convention_tools from cckit.json, or 'none'}
</research_tools>

{if mode == 'update':}
<existing_convention>
{content of existing convention file}
</existing_convention>
{end if}",
  run_in_background: false
)
```

**Step 3: Handle Research Result**

Parse the researcher's return message:
- If "## RESEARCH COMPLETE": proceed to Step 4
- If "## RESEARCH FAILED": display the failure reason and AskUserQuestion: "Research failed. Options?" with choices: ["Retry research", "Proceed with limited context", "Cancel"]. On retry: re-dispatch. On proceed: continue with whatever findings exist. On cancel: stop.

**Step 4: Present Research Summary**

Display progress announcement: "Research complete. Here's what I found:"

Present the research findings to the user in a readable format:
- Best practices as a numbered list with source attribution
- Delta test assessment summary: "{N} practices found, {N} high-delta (will teach), {N} low-delta (consistency value), {N} no-delta (Claude already knows)"
- If language-specific: present library evaluation tables (per D-15):
  - Health/soundness table
  - Feature comparison table
  - Recommendation with reasoning
  - Note: "You'll make the final library selection during preference collection."
- Common pitfalls list

Per D-33: If the research report includes an "Insufficient Coverage" section, display it and AskUserQuestion: "Research found limited information in some areas. Options?" with choices: ["Proceed with available research", "Provide additional context or URLs", "Cancel"]. If user provides URLs: note them as additional context for the generator.

**Step 5: Transition**

- **If research-first flow:** -> Read step-preferences.md via `$CLAUDE_SKILL_DIR/step-preferences.md`. Pass forward: all init variables + research_results (full report content).
- **If preferences-first flow (called after step-preferences):** -> Read step-generate.md via `$CLAUDE_SKILL_DIR/step-generate.md`. Pass forward: all init variables + research_results + user_preferences.

Title: `# Step: Preferences`

**Step 1: Determine Question Strategy**

Based on the convention area and available research results:
- If research results are available (research-first flow): derive questions from research findings -- present specific options the user might not have known about (per D-22: "research-informed questions")
- If no research results yet (preferences-first flow): ask broader questions about the user's existing preferences and style

The number and type of questions depend entirely on the convention area (per D-06: adaptive, never hardcoded). Examples of question categories that MAY apply depending on the area:
- Format preferences (e.g., for commit conventions: subject line length, body format, scope style)
- Strictness level (strict enforcement vs. flexible guidance)
- Library preferences (if language-specific and research found library options -- present the library comparison and ask user to select)
- Existing practices the user wants preserved
- Practices the user explicitly disagrees with from research

**Step 2: Adaptive Question Loop**

For each preference area relevant to this convention:

1. Present context in conversation text -- explain what the preference affects and what options exist (if research-informed, reference specific findings)
2. AskUserQuestion with concrete options (not open-ended "what do you prefer?" -- give specific choices derived from research or common patterns)
3. Record the user's selection
4. If the selection conflicts with a best practice from research (per D-23): inform the user of the trade-off. Present in conversation text: "Note: your preference for {X} differs from the commonly recommended {Y}. The trade-off is {trade-off}." Then AskUserQuestion: "Keep your preference?" with options: ["Yes, keep {X}", "Switch to {Y}", "Tell me more"]. Never silently override.

Continue until all relevant preference areas are covered. Do NOT ask a fixed number of questions -- some areas need 2-3 questions, others might need 8-10.

After collecting preferences, present a summary: "Here's what I'll use to generate your convention:" followed by a bulleted list of all collected preferences.

AskUserQuestion: "Preferences look correct?" with options: ["Yes, proceed to generation", "I want to change something", "Add another preference"]. Loop if user wants changes.

**Step 3: Transition**

- **If research-first flow (research already done):** -> Read step-generate.md via `$CLAUDE_SKILL_DIR/step-generate.md`. Pass forward: all variables + research_results + user_preferences.
- **If preferences-first flow (research not yet done):** -> Read step-research.md via `$CLAUDE_SKILL_DIR/step-research.md`. Pass forward: all variables + user_preferences (researcher will use preferences to focus research). After research completes, research step transitions to step-generate.md.

Pass forward to generate step: `area`, `lang`, `publisher`, `convention_tools`, `resolved_path`, `mode`, `research_results` (full report), `user_preferences` (structured summary of all preferences collected).

**Create skills/convention/step-generate.md:**

Title: `# Step: Generate`

This step dispatches the convention-generator agent, shows the preview, and handles the approval cycle. It is the convergence point for both research-first and preferences-first flows.

**Step 1: Prepare Generator Input**

Assemble the input for the convention-generator agent from the collected context:
- `research_results`: full research report from step-research
- `user_preferences`: structured preferences from step-preferences
- `output_path`: resolved path from step-init
- Determine convention type: if `lang` is nil/absent -> "base", otherwise -> "language-specific ({lang})"
- Read docs/CONVENTIONS.md for the generator to reference

Build delta test instructions (always provided):
```
For each candidate rule, apply the delta test:
- Would Claude already follow this rule without being told? If yes -> REMOVE
- Does the user want behavior different from Claude's default? If yes -> KEEP
- Would Claude's behavior vary across sessions without this rule? If yes -> KEEP
Report removed rules and reasons in your generation report.
```

Build tech-neutrality check (base conventions only, per D-21):
```
Test every rule: 'Could a Rust CLI, Go API, Python ML pipeline, and
documentation site all follow this rule?' If no -> belongs in language-specific.
```

**Step 2: Dispatch Generator Agent**

Display progress announcement (per D-07): "Generating convention file..."

```
Agent(
  subagent_type: "convention-generator",
  prompt: "<objective>
Generate a {base_or_lang} convention file for the '{area}' area{' targeting ' + lang if lang else ''}.
</objective>

<convention_spec>
Read docs/CONVENTIONS.md for the authoritative convention file format specification.
</convention_spec>

<research_results>
{full research report}
</research_results>

<user_preferences>
{structured preferences}
</user_preferences>

<delta_test_instructions>
{delta test criteria as above}
</delta_test_instructions>

{if base convention:}
<tech_neutrality_check>
Test every rule: 'Could a Rust CLI, Go API, Python ML pipeline, and
documentation site all follow this rule?' If no -> belongs in language-specific.
</tech_neutrality_check>
{end if}

<output_path>{resolved_path}</output_path>
<mode>create</mode>",
  run_in_background: false
)
```

**Step 3: Handle Generator Result**

Parse the generator's return:
- If "## GENERATION COMPLETE": proceed to Step 4
- If "## GENERATION FAILED": display the failure reason and AskUserQuestion: "Generation failed. Options?" with choices: ["Retry", "Cancel"]. On retry: re-dispatch with same inputs.

**Step 4: Orchestrator Light Review (per D-20)**

Perform a lightweight post-generation check:
- Read the generated file from the output path
- Verify frontmatter format:
  - Base: has `description`, no `paths`
  - Lang-specific: has `description`, has `paths: {glob}` (unquoted), has `alwaysApply: false`
- Verify footer comment exists: `<!-- Generated by /convention, {timestamp} -->`
- For base conventions: quick scan for obvious technology-specific terms (specific language names, framework names, tool names) -- flag if found
- Parse the generation report for the delta test results

If any issues found: note them but do NOT block. Present them as warnings alongside the preview.

**Step 5: Handle Empty Convention (per D-32)**

Check the generation report's "Rules Included" count:
- If count is 0 (all rules filtered by delta test):

Present in conversation text:
"All candidate rules for the '{area}' convention were filtered by the delta test. Claude already follows these practices by default, so a convention file would not add value."

Display the list of filtered rules and their removal reasons from the generation report.

AskUserQuestion: "All rules filtered. Options?" with choices: ["Force-create anyway (keep all rules)", "Skip this convention (no file created)", "Adjust preferences and re-generate"]

- Force-create: re-dispatch generator with delta test disabled (add `<force_create>true</force_create>` tag)
- Skip: delete the generated file (if it was written), report "Convention skipped -- not needed."
- Adjust: loop back to step-preferences

**Step 6: Preview and Approval (per D-24)**

Display progress announcement: "Preview:"

Read the generated convention file and display it in a markdown code block.

Also display the generation report summary:
- Rules included: {count}
- Rules removed by delta test: {count} (list with reasons)
- Tech-neutrality results (if base): passed {count}, failed {count}
- Confidence: {level}
- Any warnings from the orchestrator review (Step 4)

AskUserQuestion: "Convention preview" with options: ["Approve and write", "Request changes", "Cancel"]

- **Approve**: confirm the file is written (generator already wrote it in Step 2). Display: "Convention written to {resolved_path}". Done.
- **Request changes** (per D-26): present in conversation text "What would you change?" Then AskUserQuestion to collect feedback (free-form). Re-dispatch generator with the same inputs PLUS the `<feedback>` tag containing user's feedback. Loop back to Step 2 (re-generation).
- **Cancel**: delete the generated file. Report "Convention cancelled."

**Create skills/convention/step-update.md:**

Title: `# Step: Update`

This step handles updating an existing convention. It is reached when step-init detects the target file already exists.

**Step 1: Read Existing Convention**

Read the existing convention file from the resolved path.
Display to user: "Found existing convention at {resolved_path}. Here's the current content:" followed by the file content in a code block.

**Step 2: Update Strategy Choice (per D-08)**

Analyze the existing convention and present options:

"How would you like to update this convention?

**Full rewrite:** Re-research the area, re-collect preferences, and generate a completely new version. The delta test is re-validated. Best when conventions have drifted significantly or you want a fresh start.

**Surgical edit:** Modify specific rules only. Add new rules, change existing ones, or remove rules you no longer want. Best for targeted changes."

AskUserQuestion: "Update approach?" with options: ["Full rewrite", "Surgical edit"]

**Step 3a: Full Rewrite Path**

If user chooses full rewrite:
- This is essentially the create flow. Transition to the create workflow:
  - AskUserQuestion for flow choice (research-first or preferences-first) -- same as step-init Step 7
  - Route to step-research.md or step-preferences.md accordingly
  - The generator receives `<mode>create</mode>` and overwrites the file
  - step-generate handles preview/approval as normal

Pass the `<existing_convention>` content to the researcher so it can focus on what's changed in the ecosystem since the convention was written.

-> Read step-research.md or step-preferences.md via `$CLAUDE_SKILL_DIR/`.

**Step 3b: Surgical Edit Path (per D-09)**

If user chooses surgical edit:

Present in conversation text: "What changes would you like to make? You can:"
- Modify existing rules (change wording, strictness, scope)
- Add new rules
- Remove rules that are no longer relevant

Use an adaptive AskUserQuestion loop to collect changes:
1. AskUserQuestion: "What would you like to change?" (free-form or specific options if the convention structure suggests them)
2. Record the requested change
3. AskUserQuestion: "Any more changes?" with options: ["Add another change", "That's all"]
4. Loop until user says "That's all"

**Step 4b: Dispatch Generator for Surgical Edit**

Display progress announcement: "Applying changes..."

```
Agent(
  subagent_type: "convention-generator",
  prompt: "<objective>
Apply surgical edits to the existing '{area}' convention{lang clause}.
</objective>

<convention_spec>
Read docs/CONVENTIONS.md for the authoritative convention file format specification.
</convention_spec>

<research_results>
Not applicable for surgical edit -- use existing convention as context.
</research_results>

<user_preferences>
{existing convention represents prior preferences}
</user_preferences>

<delta_test_instructions>
For newly added or modified rules only:
- Would Claude already follow this rule without being told? If yes -> flag for user
- Does the user want this behavior? If yes -> KEEP (user explicitly requested)
</delta_test_instructions>

<output_path>{resolved_path}</output_path>
<mode>surgical-edit</mode>

<existing_content>
{full content of existing convention file}
</existing_content>

<requested_changes>
{structured list of user's requested changes from Step 3b}
</requested_changes>",
  run_in_background: false
)
```

**Step 5b: Diff Preview and Approval (per D-25)**

Parse the generator's return. If "## GENERATION COMPLETE":

Display progress announcement: "Changes preview:"

Present a structured change report (not raw diff):
- Rules added: {list with content}
- Rules modified: {list with before -> after}
- Rules removed: {list}
- Unchanged rules: {count}

AskUserQuestion: "Update preview" with options: ["Approve changes", "Request adjustments", "Cancel update"]

- **Approve**: confirm file written. Display: "Convention updated at {resolved_path}". Done.
- **Request adjustments**: collect additional feedback via AskUserQuestion, re-dispatch generator with accumulated changes + adjustments. Loop.
- **Cancel**: revert file to original content (re-write the existing content). Report "Update cancelled."
