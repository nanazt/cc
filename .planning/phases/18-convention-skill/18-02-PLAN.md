**Create skills/convention/SKILL.md** with the following structure, modeled after skills/case/SKILL.md:

**Frontmatter** (per D-35 and 18-RESEARCH.md pattern):
- name: convention
- description: (multiline >) Research-driven convention authoring tool. Creates and updates convention files (Claude Code rules) following the cckit convention architecture. Use for: convention creation, convention update, best practice research, style preference collection.
- argument-hint: "{area} [lang]"
- allowed-tools: Read, Write, Bash, Glob, Grep, AskUserQuestion, Agent, WebSearch, WebFetch
- disable-model-invocation: true

**Body sections (in order):**

1. `<purpose>` -- Research and author convention files (Claude Code rules) that follow the cckit convention architecture (docs/CONVENTIONS.md). Create new conventions through research + user preference collection, or update existing conventions via rewrite or surgical edit. One convention per invocation. Convention area is required, language is optional.

2. `<philosophy>` -- Key principles:
   - "You are the Architect." The user provides the vision; you research, structure, and generate. Your value is turning scattered best practices into precise, actionable Claude Code rules.
   - Research before opinion. Never generate convention rules from training data alone. Always research current best practices first.
   - Delta test is non-negotiable. Every rule must earn its place. If Claude already follows a practice by default, the rule adds noise, not value.
   - User preferences override research. When the user wants something different from best practices, honor it. Inform of trade-offs, never override.
   - Conventions are directives, not documentation. Every line is an instruction to Claude. No rationale, no history, no explanations.
   - Technology neutrality for base conventions. If a rule mentions a specific language, framework, or tool, it belongs in a language-specific file.
   - Adaptive interaction. Ask as many or as few questions as the convention area requires. Never hardcode question counts. (per D-06)
   - All user-facing questions use AskUserQuestion tool. Present detailed context in conversation text, then ask via AskUserQuestion with a short one-liner.

3. `<formatting>` -- Rules for user interaction:
   - Phase announcements for progress feedback (per D-07): display "Researching {area} conventions..." before research dispatch, "Research complete. Here's what I found:" before presenting results, "Collecting your preferences..." before preference questions, "Generating convention file..." before generator dispatch, "Preview:" before showing generated content.
   - Research summaries presented as numbered lists with source attribution
   - Library evaluations presented as two tables (health/soundness + feature comparison) per D-15
   - Preference questions via AskUserQuestion with options derived from research results (per D-22)
   - Convention preview displayed in a code block for easy review
   - Update mode diffs presented as structured change report (rules added/removed/modified), not raw diff

4. `<available_agent_types>` (following case SKILL.md pattern):
   - `convention-researcher` -- Researches best practices, evaluates libraries, identifies delta test candidates. Returns structured research report. Model: opus.
   - `convention-generator` -- Generates convention files with embedded delta test and tech-neutrality validation. Handles create and surgical-edit modes. Returns generated file + generation report. Model: sonnet.

5. `<step_routing>` -- Routing table and file state detection.

   File state detection approach:
   - Parse arguments: area from $ARGUMENTS (first word), lang from $ARGUMENTS (second word, optional)
   - Read .claude/cckit.json if it exists to determine publisher mode
   - Resolve target file path based on publisher flag and area/lang
   - Check if target file exists to determine create vs update mode

   Routing table:

   | Target file exists | Mode | Route to |
   |-------------------|------|----------|
   | no | create | step-init.md -> (flow selection) -> step-research.md or step-preferences.md -> step-generate.md |
   | yes | update | step-init.md -> step-update.md |

   Step transitions:
   - step-init.md complete (create) -> AskUserQuestion for flow choice -> load step-research.md or step-preferences.md
   - step-init.md complete (update) -> load step-update.md
   - step-research.md complete -> load step-preferences.md (research-first flow)
   - step-preferences.md complete -> load step-generate.md (both flows converge)
   - step-preferences.md complete (preferences-first) -> load step-research.md -> load step-generate.md
   - step-generate.md complete -> done (file written and approved)
   - step-update.md complete -> done (file updated and approved)

   IMPORTANT: When transitioning between steps, explicitly Read the next step file using `$CLAUDE_SKILL_DIR` variable. Step files are NOT preloaded.

6. `<success_criteria>` (following case SKILL.md pattern):

   - Tech-neutrality validated (base conventions only)
   - User approved the preview (or provided feedback for re-generation)
   - Convention file written to resolved path
   - Footer comment added with ISO 8601 timestamp

**Structure (all content in markdown, no frontmatter):**

**Title:** `# Step: Init`

**Step 1: Parse Arguments**

Parse $ARGUMENTS: area = first word (required), lang = second word (optional). If area is empty or missing, report error "Convention area is required. Usage: /convention {area} [lang]" and stop. Normalize both to lowercase.

**Step 2: Read Config**

Read `.claude/cckit.json` if it exists using `cat .claude/cckit.json 2>/dev/null`. Parse JSON for: `publisher` (boolean, default false) which determines output path strategy, and `convention.tools` (string[], default []) for additional researcher tool names. If file does not exist or is invalid JSON, use defaults.

**Step 3: Resolve Output Path**

Based on publisher flag:
- publisher == true: Base path = `conventions/{area}/CONVENTION.md`, Lang path = `conventions/{area}/{lang}.md`
- publisher == false or absent: Base path = `.claude/rules/cckit-{area}.md`, Lang path = `.claude/rules/cckit-{area}-{lang}.md`

Store the resolved path for use in subsequent steps.

**Step 4: Detect Mode (Create vs Update)**

Check if the target file already exists (per D-02) using `test -f {resolved_path}`. If file exists: mode = "update". If not: mode = "create".

**Step 5: Convention Naming (Create Mode Only)**

Per D-04: Only when creating a NEW convention area (directory/prefix does not yet exist), present recommended names via AskUserQuestion. Show in conversation text: "The convention area name determines the directory (publisher) or file prefix (consumer). Common convention areas include: commit, coding, test, error, documentation, workflow, security." Then AskUserQuestion: "Convention area name?" with options including the user-provided area name plus common alternatives plus "Other (type a custom name)". If user selects "Other", accept typed response and re-resolve path. Skip this step entirely if the area already exists.

**Step 6: Handle Missing Base (Lang-Specific Only)**

Per D-30: If lang is provided but no base convention exists for this area (check `conventions/{area}/CONVENTION.md` in publisher mode or `.claude/rules/cckit-{area}.md` in consumer mode):

Present context and AskUserQuestion: "How to proceed?" with options:
1. "Create base first, then {lang}" -- set create_base_first flag
2. "Proceed as standalone {lang} convention" -- self-contained per docs/CONVENTIONS.md
3. "Cancel" -- stop

**Step 7: Flow Selection (Create Mode Only)**

Per D-05: Present workflow choice in conversation text explaining research-first vs preferences-first approaches. AskUserQuestion: "Convention workflow?" with options: ["Research first", "Preferences first"]. Store selection for routing.

**Transition section:**

When init is complete:
- Create mode, research-first: Read step-research.md via `$CLAUDE_SKILL_DIR/step-research.md`
- Create mode, preferences-first: Read step-preferences.md via `$CLAUDE_SKILL_DIR/step-preferences.md`
- Update mode: Read step-update.md via `$CLAUDE_SKILL_DIR/step-update.md`

Pass forward: area, lang (or nil), publisher flag, convention_tools list, resolved_path, mode, selected_flow, create_base_first flag (if set).
