### Pattern 1: Structural Placeholder Convention

**What:** Replace concrete domain-specific content with bracketed structural placeholders that describe the FORMAT and QUALITY BAR, not the domain content.

**When to use:** Every biased inline example, template cell, or guide text across all files.

**Convention established by CONTEXT.md decisions:**
```
Concrete (biased)             ->  Structural (neutral)
------                            ------
400 bad request                   [error outcome (ErrorName)]
Auth: any authenticated role      Caller: [who/what can invoke this]
"use mock SMTP during dev"        [implementation approach that is explicitly temporary]
User.CreateUser                   [Component.OperationName]
"all endpoints require auth"      [system-wide access policy]
returns 401                       returns [status/error code]
```

### Pattern 2: Open Format Replacement

**What:** Replace fixed category lists with open-ended prompts that let Claude adapt to the host project.

**When to use:** The 6 fixed side effect categories in step-discuss.md (D-08).

**Before:** 6 hardcoded categories (Domain events, Related entity updates, Cache mutations, Audit/logging, Notifications, External system calls)
**After:** Open format -- "list expected side effects for this operation" with guidance on what constitutes a side effect, without prescribing categories.

### Pattern 3: Structural Rule Teaching

**What:** Teach the rule (format + quality bar) instead of teaching by example.

**When to use:** Error naming convention (D-10), Expected Outcome guidance (D-05, D-07).

**Principle:** The current files teach via concrete examples ("400 Bad Request (DuplicateEmail)"). The replacement teaches the structural rule itself: `"[error description] (ErrorName)"` with the criterion "specific enough that a test can assert against it."

### Pattern 4: Global + Selective Inline Adaptation Instruction

**What:** One global adaptation statement in SKILL.md + selective per-site annotations only where the structural placeholder is non-obvious.

**When to use:** D-19 hybrid approach. Most placeholders like `[OperationName]` are self-explanatory and rely on the global instruction. Error naming convention and Expected Outcome guidance get inline annotations because the gap between placeholder and correct domain-specific output is larger.

### Anti-Patterns to Avoid

- **Neutral-sounding concrete examples:** Replacing "JWT token" with "session token" still anchors Claude to token-based auth. The principle is structural placeholders, not domain rotation.
- **Category shifting:** Replacing "Domain events" with "State change notifications" still imposes a fixed category. Open format means no categories.
- **Protocol showcase framing:** Even when individual examples are neutral (e.g., "credentials invalid"), if they appear in a block framed as "here are examples across protocols," the framing itself is biased (D-11).
- **Over-annotation:** Adding inline adaptation notes everywhere dilutes their signal. Only non-obvious sites get annotations (D-19).

## Common Pitfalls

### Pitfall 1: Accidental Bias Reintroduction

**What goes wrong:** When writing structural placeholders, the author unconsciously chooses examples or guide text that still assume a specific architecture (e.g., "the function returns an error" assumes synchronous call/response).
**Why it happens:** Deep familiarity with web service patterns makes them feel "universal" when they are not.
**How to avoid:** Apply the CLAUDE.md neutrality test for each replacement: "Could any project -- regardless of type, language, or domain -- install this and use it without editing the plugin?" Test mentally against: CLI tool, desktop app, library, OS component, non-code project.
**Warning signs:** Concrete nouns (endpoint, route, request, response, token, session, cookie) appearing in supposedly neutral text.

### Pitfall 2: Over-Abstraction Making Guide Text Useless

**What goes wrong:** Placeholders become so abstract that Claude cannot understand what to produce. `[thing that happens]` is not a useful guide.
**Why it happens:** Pushing too hard on neutrality without maintaining the "specific enough that a test can assert" criterion (D-02).
**How to avoid:** Each placeholder must communicate: (1) the structural role (error outcome, side effect, caller identity) and (2) the quality bar (specificity level). "Be specific enough that a test can assert against it" is the anchor. Structural placeholder wording should describe WHAT KIND of information goes here, not just that information goes here.
**Warning signs:** Placeholders that contain no hint about what category of content they expect.

### Pitfall 3: Missing a Biased Site

**What goes wrong:** One or more biased examples survive the rewrite because they were not identified in CONTEXT.md's exhaustive list.
**Why it happens:** The CONTEXT.md identifies biased sites by line number as of the time the discussion occurred. If Phase 12 changed line numbers, references may be stale.
**How to avoid:** Use the CONTEXT.md list as a starting guide but also do a full read of each file, looking for the bias definition: "assumes a specific communication protocol, storage mechanism, auth mechanism, architecture pattern, or project type" (D-15). The comprehensive list in CONTEXT.md's "Biased inline examples identified for replacement" section is the canonical inventory, but verify line numbers against current file state.
**Warning signs:** Any remaining instance of: HTTP status codes (400, 401, 500, 201, 204), protocol names (gRPC, REST, HTTP), specific auth tech (JWT, OAuth, WebAuthn, passkey), specific infrastructure (Redis, SMTP, JSONB), specific architecture (middleware, endpoint, route, cookie, session/ceremony).

### Pitfall 4: Breaking the Canonical Flow Diagram's Readability

**What goes wrong:** The canonical flow diagram in SKILL.md becomes hard to read when HTTP status codes are replaced with longer structural placeholders, breaking ASCII alignment.
**Why it happens:** `400 bad request` is 15 chars; `[error outcome (ErrorName)]` is 27 chars. Flow diagram branches have alignment constraints.
**How to avoid:** Keep placeholder text in the flow diagram short. The flow diagram is a FORMAT guide -- it demonstrates diagram structure, not error naming. Failure branches can use abbreviated forms: `F1: failure case -> [error outcome (ErrorName)]`. Test the visual alignment after editing.
**Warning signs:** Flow diagram lines exceeding ~80 chars, misaligned branch indicators.

### Pitfall 5: Inconsistent Treatment Across Files

**What goes wrong:** The same concept (e.g., error naming) is neutralized with different placeholder wording in different files, creating contradictory guidance for Claude.
**Why it happens:** Files are edited independently without cross-referencing.
**How to avoid:** Establish the canonical placeholder vocabulary in SKILL.md first (the global adaptation instruction per D-19), then reference/reuse the same terms in other files. The error naming structural rule `[error description] (ErrorName)` from D-10 must be worded identically wherever it appears.
**Warning signs:** Different bracketed expressions for the same concept across files (e.g., `[error (Name)]` in one file vs `[error description (ErrorName)]` in another).
