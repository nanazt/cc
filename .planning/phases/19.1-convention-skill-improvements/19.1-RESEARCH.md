## Rule vs Skill Judgment Criteria (D-03 Research)

This section fulfills D-03: defining the criteria the skill uses to judge whether a convention should be a Rule or a Skill.

### Decision Framework

The judgment follows from Claude Code's loading behavior [CITED: code.claude.com/docs/en/skills]:

| Property | Rule (.claude/rules/) | Skill (.claude/skills/) |
|----------|----------------------|------------------------|
| Loading | Always in context (or path-scoped) | Description always in context; body loaded on-demand |
| Context cost | Full content every session | Description only (~1 line) until invoked |
| Tool access | None | Optional via allowed-tools |
| Invocation | Automatic (no user action) | User `/name` or Claude auto-invoke |
| Best for | Passive guidance Claude must always follow | Context-specific guidance triggered by events or user |

### Three-Criteria Test

The skill's step-init (or step-research) should apply these criteria in order:

**Criterion 1: Event-Driven Loading** -- Does this convention primarily apply during a specific action or event (commit, PR, deploy, test run) rather than continuously?
- YES -> Skill candidate. Context cost is wasted when the event is not happening.
- NO -> Rule candidate. Continuous applicability justifies always-on loading.

**Criterion 2: Tool Access Need** -- Would this convention benefit from Claude having tool access (Read, Bash, Grep) when applying it?
- YES -> Skill. Rules cannot grant tool access. [VERIFIED: Claude Code docs -- rules have no allowed-tools field]
- NO -> Continue to next criterion.

**Criterion 3: Context Size** -- Is the convention content large enough (>50 lines) that always-loading it wastes significant context?
- YES -> Skill candidate. On-demand loading preserves context budget.
- NO -> Rule. Small conventions have negligible context cost.

**Override: User Preference** -- If the user explicitly requests a specific type, honor it regardless of criteria. Document the trade-off but do not override.

### Recommendation for Existing Convention Areas

| Convention Area | Recommended Type | Reasoning |
|----------------|-----------------|-----------|
| commit | Skill | Event-driven (only during commits); hook already provides event injection |
| coding | Rule | Continuous applicability; applies to all code edits |
| test | Rule | Continuous when writing tests; path-scoped suffices |
| error | Rule | Continuous pattern guidance; path-scoped suffices |
| documentation | Rule | Continuous when writing docs |
| workflow | Skill | Event-driven (PR, branch operations); may need tool access for validation |
| security | Rule | Must always be in context; security guidance should never be skippable |

[ASSUMED] -- These recommendations are based on the criteria analysis. The user may have different priorities for specific areas. The criteria should be presented to the user for confirmation when authoring each convention.

1. Read the host project's CLAUDE.md (or .claude/CLAUDE.md)

3. Conflict detection is text-based heuristic, not semantic -- look for same topic area with different instruction

**Practical implementation:** This is a lightweight scan, not an AI analysis. The orchestrator reads CLAUDE.md, identifies sections that overlap with the convention area (e.g., "Commit Conventions" section in CLAUDE.md for a commit convention), and flags the overlap. The user decides.

## Common Pitfalls

### Pitfall 1: Skill Install Path vs Rule Install Path
**What goes wrong:** D-05 specifies skill consumer path as `.claude/skills/cckit-{area}.md` but Claude Code skills are directory-based (`skill-name/SKILL.md`).
**Why it happens:** Rules are flat files in `.claude/rules/`; skills require a directory structure.
**How to avoid:** Consumer skill install path must be `.claude/skills/cckit-{area}/SKILL.md`, not `.claude/skills/cckit-{area}.md`. Flag this discrepancy to the user during planning.
**Warning signs:** Installed skill not discovered by Claude Code.

### Pitfall 2: Generator Scope Creep
**What goes wrong:** Generator agent now handles convention + hook + test + fixtures -- four artifact types. Quality drops across all.
**Why it happens:** D-09 specifies single-agent generation for coherence, but the prompt becomes very long.
**How to avoid:** Structure the generator prompt clearly with separate sections for each artifact. Provide reference implementations (existing validate.sh, test.sh, fixtures/) as exemplars in the prompt.
**Warning signs:** Hook scripts that don't match convention rules; test fixtures that don't test the right things.

### Pitfall 3: .state/ Cleanup on Error
**What goes wrong:** If generation fails or user cancels mid-flow, `.state/` directory persists with stale data.
**Why it happens:** D-15 says cleanup after "successful generation" but does not specify error/cancel paths.
**How to avoid:** Explicit cleanup in all exit paths: cancel, failure, and success. The orchestrator (step-generate) should handle cleanup in its Cancel and Failure branches.
**Warning signs:** Stale `.state/` directories in convention areas.

### Pitfall 4: writing-skills Skill Access from Subagent
**What goes wrong:** Generator agent cannot invoke `Skill("superpowers:writing-skills")` because it does not have the Skill tool in its allowed-tools.
**Why it happens:** Generator agent's tools are `Read, Grep, Glob, Write` -- no Skill tool.
**How to avoid:** Orchestrator reads writing-skills content and injects it into generator prompt. Do not add Skill tool to generator.
**Warning signs:** Generator produces skill-type files with incorrect format.

### Pitfall 5: Delta Tag Parsing Fragility
**What goes wrong:** Generator fails to parse `[DELTA:new]` and `[HOOK:yes]` tags from researcher output.
**Why it happens:** Free-form markdown is fragile to parse. Tags may appear in unexpected positions.
**How to avoid:** Define exact tag format in both researcher output contract and generator input contract. Tags must appear on the same line as the practice name, enclosed in square brackets.
**Warning signs:** Generator treats all rules as new, ignoring known/varies classification.

### Pitfall 6: Hook Discovery Path Length
**What goes wrong:** Unified discovery checks 4 paths but area name may differ between publisher and consumer formats.
**Why it happens:** Publisher uses `conventions/{area}/` directory; consumer uses `cckit-{area}` prefix or directory.
**How to avoid:** Discovery path must be parameterized by area name, not hardcoded. The hook template generated by the skill should include the area name variable.
**Warning signs:** Hook works in publisher mode but not consumer mode or vice versa.
