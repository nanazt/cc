# Phase 19.1 Plan 04: Commit Convention Migration and step-update Hook Sync Summary

**One-liner:** Migrated commit convention from rule format (CONVENTION.md) to skill format (SKILL.md) with unified 4-path hook discovery, and updated step-update to auto-regenerate hooks on both surgical edits and full rewrites.

## What Was Done

### Task 1: Migrate commit convention to SKILL.md and update hook discovery (c1e50e2)

Created `conventions/commit/SKILL.md` with skill-type frontmatter (`name: cckit-commit`, `description:`, `user-invocable: false`) and identical body content from the former `CONVENTION.md`. The body covers Format, Scope, Subject Line, Body, No Project Management References, Breaking Changes, Issue References, and Atomic Commits sections.

Deleted `conventions/commit/CONVENTION.md`.

Updated `conventions/commit/hooks/validate.sh`:
- Changed discovery from an if/elif 2-path check to a `for path in` 4-path loop
- Order: `SKILL.md` → `CONVENTION.md` → `.claude/skills/cckit-commit/SKILL.md` → `.claude/rules/cckit-commit.md`
- Updated comment from "publisher then consumer" to "unified: skill first, then rule"

Added comment to `conventions/commit/hooks/test.sh` documenting that `valid-commit.json` (using `__CWD__`) now discovers `SKILL.md` first after migration.

All 16 hook tests pass after migration.

### Task 2: Update step-update with hook sync, skill-type handling, and .state/ integration (7207b3d)

Updated `skills/convention/step-update.md` with:

**Step 1 additions:**
- Artifact type detection from resolved_path (SKILL.md → "skill", CONVENTION.md → "rule", consumer paths)
- Hook existence check for both publisher and consumer modes
- Display note when hooks exist: automatic regeneration on convention update

**Step 2 (strategy choice) additions:**
- Both "Full rewrite" and "Surgical edit" descriptions mention hook regeneration when `hooks_exist` is true

**Step 3a (full rewrite) additions:**
- Pass `artifact_type`, `hooks_exist`, `state_dir` through both research-first and preferences-first paths
- Note explaining that step-generate handles hook regeneration when `hook_confirmed` is true

**New Step 4c: Hook Regeneration on Surgical Edit:**
- Dispatches generator in `mode=hook-regenerate` after surgical edit succeeds
- Passes `hook_confirmed: true`, `hook_rules` (mechanically verifiable rules), and `hook_trigger` (from existing validate.sh)
- Failure handling: retry/skip options; skip leaves old hooks with an out-of-sync warning

**Step 5b (change report) additions:**
- Hook status section: "Regenerated to match updated rules" or "NOT updated -- may be out of sync"
- Hook diff display (optional, bundled with convention approval — no separate gate)

**Transition additions:**
- Display "Hooks regenerated at {hook_path}" and test run command when hooks were regenerated
- Display warning when hooks were NOT regenerated
