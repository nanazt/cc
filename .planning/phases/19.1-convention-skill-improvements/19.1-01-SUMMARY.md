# Phase 19.1 Plan 01: Architecture Foundation Summary

**One-liner:** Rule/skill binary classification with three-criteria type detection, .state/ persistence init, and unified hook discovery path — establishing the architecture contracts that Plans 02–04 implement against.

## What Was Built

### Task 1: docs/CONVENTIONS.md — Rule/Skill Binary Classification

Updated the convention architecture specification with:

- **Overview section:** Changed "What conventions are" from "Claude Code rules" to "Claude Code rules or skills" with a new comparison table (rule-type vs skill-type properties)
- **Source Structure:** Added `SKILL.md` row to the file table alongside `CONVENTION.md`; added note that each area has either CONVENTION.md or SKILL.md, never both; updated canonical example to show commit as skill-type and coding as rule-type
- **Convention File Format:** Added new "Skill-Type Convention (On-Demand)" subsection with example frontmatter (`name`, `description`, `user-invocable: false`) and explanation of when `allowed-tools` applies
- **Naming and Installation:** Added skill-type row to source-to-installed mapping table; added note about directory structure requirement for skills vs flat files for rules
- **Hook Architecture:** Added "Unified Convention Discovery" subsection with the 4-path discovery order (publisher skill → publisher rule → consumer skill → consumer rule, first match wins)
- **Anti-Patterns:** Replaced "Convention-as-skill" row (no longer an anti-pattern) with "Wrong artifact type" guidance pointing to the three-criteria test
- **Summary Table:** Added "Artifact type decision" and "Skill frontmatter fields" rows; updated "Convention format" and "Base file name" values; updated "Installed prefix" to note flat vs directory structure

### Task 2: skills/convention/SKILL.md — Orchestrator Updates

- **description:** "Claude Code rules" → "Claude Code rules or skills"
- **purpose:** Added "Output is either a rule (always-loaded, passive) or a skill (on-demand, context-aware) — determined during init."
- **step_routing file state detection:** Updated to check BOTH CONVENTION.md and SKILL.md for create/update mode detection; clarified that resolved_path for create mode depends on artifact type from step-init
- **philosophy:** Added "Right artifact type for the job" principle with event-driven vs continuous guidance and reference to three-criteria test
- **success_criteria:** Added "Artifact type (rule or skill) determined correctly" and "Step output persisted to .state/ directory" criteria

### Task 2: skills/convention/step-init.md — Artifact Type Detection and .state/ Init

- **Step 3 restructured** into Step 3a (preliminary path check for both filenames) — mode detection only; final path deferred to Step 7b
- **Step 4 updated** to check both CONVENTION.md and SKILL.md for base convention mode detection (publisher and consumer modes)
- **Step 6 updated** to check both possible base convention filenames when determining if base exists
- **Step 7a added** — Artifact Type Detection (create mode only, skip for update or lang): applies three-criteria test (event-driven loading, tool access need, context size), presents recommendation, asks via AskUserQuestion with Rule/Skill options, stores `artifact_type`
- **Step 7b added** — Final Path Resolution: resolves definitive output path using `artifact_type` for publisher and consumer modes
- **Step 8 added** — Initialize .state/ Directory (create mode only): creates state directory, writes `init.json` with all step variables (area, lang, publisher, convention_tools, resolved_path, mode, artifact_type, selected_flow, create_base_first)
- **Transition updated** to include `artifact_type` and `state_dir` in values passed forward
