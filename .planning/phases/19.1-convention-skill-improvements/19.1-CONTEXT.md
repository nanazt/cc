<decisions>
## Implementation Decisions

### Artifact Type Selection
- **D-01:** Convention output is either a **Rule** (always-loaded, passive guidance) or a **Skill** (on-demand loaded, context-aware). This is a binary classification. Phase 17 D-01 and Phase 18 D-31 ("conventions = rules") are superseded — conventions can now be either rules or skills.
- **D-02:** **Hook is an independent decision** — determined separately from rule/skill type. A convention can be rule-only, skill-only, rule+hook, or skill+hook.
- **D-03:** Criteria for rule vs skill judgment are **delegated to the phase researcher** to investigate and define. The discuss phase establishes the architecture, not the specific heuristics.

### Convention File Naming
- **D-04:** File type determines filename: Rule-type → `CONVENTION.md`, Skill-type → `SKILL.md`. Language-specific → `{lang}.md` (unchanged). This makes type visible from the filesystem without reading frontmatter.
- **D-05:** Consumer install paths: rule → `.claude/rules/cckit-{area}.md`, skill → `.claude/skills/cckit-{area}.md`.

### Skill-Type Convention Format
- **D-06:** Skill-type conventions are **full skills with allowed-tools** when tool access adds value (e.g., Read for context-aware guidance). Not all skill-type conventions need tools — some are skills purely for on-demand loading efficiency.
- **D-07:** Generator agent **invokes writing-skills superpowers skill** when generating skill-type output to ensure proper skill file format. Rule-type generation uses existing approach.

### Hook Generation — Integrated into step-generate
- **D-08:** No separate step-hook.md. Hook generation is **integrated into step-generate**. Hooks are part of the convention and generated together for coherence.
- **D-09:** **Generator agent produces all artifacts**: convention file (rule or skill) + hook script + test script + test fixtures. Single agent for consistency between convention rules and hook validation logic.
- **D-10:** Hook necessity is determined by **Researcher → user confirmation**. Researcher identifies mechanically verifiable rules during research and recommends hook. User confirms in preferences step.

### Delta Transfer Format
- **D-11:** Research→Generator delta transfer uses **structured markdown** with per-rule classification tags.
- **D-12:** Three-class delta taxonomy: `[DELTA:new]` (Claude doesn't know), `[DELTA:known]` (Claude already follows), `[DELTA:varies]` (inconsistent across sessions). Style divergence is handled in preferences step, not by researcher.
- **D-13:** Each rule also tagged with `[HOOK:yes]` or `[HOOK:no]` for mechanical verifiability. Generator uses both tags to determine what to include and whether to generate hooks.

### Variable Persistence
- **D-14:** Step output is written to **`.state/` directory** inside the convention output directory (e.g., `conventions/{area}/.state/`). Each step writes its output: `init.json`, `research.md`, `preferences.json`.
- **D-15:** `.state/` is **cleaned up after successful generation**. It exists only during the authoring session.

### CLAUDE.md Conflict Detection
- **D-16:** Orchestrator light review (in step-generate) **reads the host project's CLAUDE.md** and cross-references generated convention rules. On conflict: **warning + user choice** (convention priority / CLAUDE.md priority / keep both).

### Update-Mode Hook Sync
- **D-17:** When convention is updated (via step-update), Generator **automatically regenerates hook + test** alongside the convention. No separate sync step needed — this follows from D-08 (hooks integrated into step-generate).

### Post-Generation Verification
- **D-18:** Orchestrator runs a **preferences checklist** after generation. Reads `preferences.json` from `.state/` and compares each preference against generated output. Mismatches trigger regeneration with correction instructions.

### convention_tools Config
- **D-19:** step-init reads `convention.tools` from `.claude/cckit.json` and stores in `.state/init.json`. step-research includes tool names in researcher agent prompt so the agent can use configured MCP tools.

### Hook Convention Discovery Path
- **D-20:** Hook scripts use **unified discovery** that checks both rule and skill locations: `conventions/{area}/SKILL.md` → `conventions/{area}/CONVENTION.md` → `.claude/skills/cckit-{area}.md` → `.claude/rules/cckit-{area}.md`. First match wins.

### Commit Convention Migration
- **D-21:** The existing commit convention (`conventions/commit/CONVENTION.md`) is **converted to skill-type** (`conventions/commit/SKILL.md`) in Phase 19.1. Content remains identical; format changes to skill frontmatter.

### Architecture Documentation
- **D-22:** `docs/CONVENTIONS.md` receives a **full update** in Phase 19.1 reflecting: rule/skill binary classification, delivery mechanism (always-on vs event-driven), updated hook architecture with unified discovery, source structure with SKILL.md, and anti-pattern updates.

### Scope Expansion
- **D-23:** Phase 19.1 ROADMAP scope is **updated** to reflect discussion outcomes. Original 9 items expanded to include: docs/CONVENTIONS.md update, commit convention migration, hook discovery path, writing-skills integration, and filename convention.

<specifics>
## Specific Ideas

- User emphasized that skills are NOT only user-invocable — AI can invoke them too. This is why skill-type conventions work: Claude sees the description and can invoke when relevant, even without user typing `/name`.
- Hook injection via additionalContext is already event-driven loading. The skill-type architecture formalizes this: description always in context (lightweight), body loaded only when needed.
- The delta transfer format (`[DELTA:new/known/varies]` + `[HOOK:yes/no]`) ensures researcher's classification work is preserved through to the generator, not lost in free-text interpretation.
- Generator producing convention + hook + test in one pass ensures tight coupling: convention rules and hook validation logic can't drift apart.
- writing-skills superpowers skill should be invoked by Generator for skill-type output to guarantee proper Claude Code skill format.

</specifics>
