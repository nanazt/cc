## Summary

This phase delivered the `/convention` skill and its multi-step flow (`step-init`, `step-research`, `step-preferences`, `step-generate`, `step-update`), two subagents (`convention-researcher`, `convention-generator`), the first concrete convention (`conventions/commit/SKILL.md`), and a full hook suite (`validate.sh`, `test.sh`, fixtures). `docs/CONVENTIONS.md` defines the architecture all of this implements.

The implementation is internally coherent in the happy path. The cross-references between steps (init → research → preferences → generate) are correct: state is passed explicitly via `.state/` files, `$CLAUDE_SKILL_DIR` transitions are correct, and the SKILL.md → CONVENTION.md discovery order is consistent throughout.

Four warnings were identified, all logic gaps that could cause incorrect behavior in specific flows. Four info items are consistency and documentation quality issues.

## Warnings

### WR-01: Installer discovery algorithm omits SKILL.md installation branch

**File:** `docs/CONVENTIONS.md:230-240`
**Issue:** The pseudocode discovery algorithm only handles `CONVENTION.md → .claude/rules/cckit-{area}.md` installation. It has no branch for `SKILL.md → .claude/skills/cckit-{area}/SKILL.md`. A future installer built from this spec would silently skip skill-type conventions during installation.

The summary table at line 203-207 and the installed structure example at line 209-222 both correctly show the SKILL.md → `cckit-{area}/SKILL.md` mapping — but the pseudocode algorithm that any implementer would follow does not encode it.

**Fix:** Add a `SKILL.md` branch to the discovery algorithm:
```
For each subdirectory in conventions/:
  area_name = directory name
  If CONVENTION.md exists:
    -> install as .claude/rules/cckit-{area_name}.md
  If SKILL.md exists:
    -> install as .claude/skills/cckit-{area_name}/SKILL.md
  For each {lang}.md file:
    -> install as .claude/rules/cckit-{area_name}-{lang}.md
  If hooks/ directory exists:
    For each script in hooks/:
      -> install as .claude/hooks/cckit-{area_name}-{script_name}
```

Also update line 242 to include `SKILL.md` alongside `CONVENTION.md` in the presence detection sentence.

### WR-02: `state_dir` is undefined in update mode full-rewrite path

**File:** `skills/convention/step-update.md:104-127`
**Issue:** In update mode, step-init explicitly skips Step 8 ("Skip this step entirely if `mode == 'update'`"), so `state_dir` is never initialized. Step-update's full-rewrite path passes `state_dir` forward to step-research (line 111) and step-preferences (line 123) with the hedge "if .state/ was initialized."

The problem is step-research writes `{state_dir}/research.md` (step-research.md line 167), and step-update Step 4c reads `{state_dir}/research.md` when regenerating hooks (step-update.md line 262). If `state_dir` is undefined or empty, the `cat {state_dir}/research.md` silently produces nothing and hook regeneration proceeds without research context.

The note at line 127 says "step-research and step-preferences will initialize .state/" but step-research does not initialize `.state/` — only step-init does.

**Fix:** In step-update.md Step 3a, before transitioning to step-research, explicitly initialize `state_dir` for update-mode rewrites:

```
Before transitioning to step-research for a full rewrite, initialize state_dir:
- Publisher mode: state_dir = conventions/{area}/.state/
- Consumer mode:  state_dir = .convention-state/{area}/
- Run: mkdir -p {state_dir}
Write an update-mode init.json with mode="update" and the artifact_type detected from the existing file.
```

Remove the hedge phrase "(if .state/ was initialized)" from the pass-forward lists — `state_dir` must always be defined before transitioning.

Update the example code block to include the `INPUT=$(cat)` capture at the top, and show the convention injection `jq -Rs` pattern at the end.

### WR-04: `validate.sh` does not handle `git commit` with heredoc `COMMIT_EDITMSG` pattern

**File:** `conventions/commit/hooks/validate.sh:39-45`
**Issue:** The message extraction relies on `-m "..."` or `-m '...'` patterns in the command string. A `git commit` without `-m` (which opens an editor, or uses `-F` with a file, or a heredoc) will leave `MSG` empty and skip all mechanical validation. The hook then proceeds to convention injection — allowing the commit — without validating the message format.

This is a partial validation gap: users who write commit messages via `git commit -F msg.txt` or `git commit` (editor) bypass format checks. These are common CI patterns.

The test suite has no fixture for `git commit` without `-m`, leaving this untested.

**Fix (option A — document the limitation):** Add a comment in `validate.sh` noting the limitation:
```sh
# NOTE: Mechanical validation requires -m "..." syntax. Commits using
# -F, --file, or interactive editor bypass format validation and receive
# convention injection only.
```

Add a test fixture:
```json
{"tool_name":"Bash","tool_input":{"command":"git commit --no-edit"},"cwd":"__CWD__"}
```
With `assert_allow` (not `assert_deny`) to document and lock in the current behavior.

**Fix (option B — expand extraction):** Add extraction from `--file` and `COMMIT_EDITMSG` file paths. More complex; deferred to a later iteration.

### IN-02: `gsd-tools commit` path skips all mechanical format validation

**File:** `conventions/commit/hooks/validate.sh:39-45`, `conventions/commit/hooks/test.sh:88-91`
**Issue:** The `gsd-tools commit` command pattern uses positional arguments (`gsd-tools.cjs commit "feat(parser): add support" --files ...`) rather than `-m "..."`. The sed extraction returns empty `MSG`, so no format checks run. The `assert_allow` + `assert_has_context` tests for `gsd-tools-commit.json` verify that convention injection works, but do not cover format validation for this path.

This is a deliberate design choice (gsd-tools is trusted to produce valid messages) but it is undocumented. A reader of the test suite may incorrectly conclude that format validation also runs for the gsd-tools path.

**Fix:** Add a comment to `validate.sh` above the `gsd-tools` case arm explaining the skip:
```sh
# gsd-tools commit: convention injection only; gsd-tools validates
# its own output before invoking the commit. No -m flag means MSG
# extraction returns empty — mechanical validation skipped.
```
Add a comment to `test.sh` alongside the gsd-tools allow tests noting this limitation.

### IN-03: Hook discovery path ordering inconsistency in CONVENTIONS.md vs validate.sh

**File:** `docs/CONVENTIONS.md:313-318` vs `conventions/commit/hooks/validate.sh:22-28`
**Issue:** `docs/CONVENTIONS.md` lists the unified discovery order as:
1. `$CWD/conventions/{area}/SKILL.md` (publisher, skill-type)
2. `$CWD/conventions/{area}/CONVENTION.md` (publisher, rule-type)
3. `$CWD/.claude/skills/cckit-{area}/SKILL.md` (consumer, skill-type)

`validate.sh` lines 22-28 implement exactly this order. This is consistent.

However, `step-generate.md` (lines 139-144) describes the hook discovery order without the `SKILL.md`-first constraint — it says "4 paths per docs/CONVENTIONS.md" but doesn't call out the ordering explicitly. The convention-generator agent's quality gate (line 344) checks only that "unified discovery path" is used, not that the order matches the spec.

**Fix (minor):** In `step-generate.md` Step 8a, replace "Discover the convention file using unified discovery (4 paths per docs/CONVENTIONS.md)" with a numbered list that exactly mirrors CONVENTIONS.md's order. This makes the ordering contractual for generated hooks, not just for the reference implementation.

### IN-04: `step-update.md` does not initialize `.state/` in surgical-edit path

**File:** `skills/convention/step-update.md:132-233`
**Issue:** The surgical edit path (Step 3b → Step 4b) never initializes `.state/`. When `hooks_exist` is true, Step 4c reads `{state_dir}/research.md` if available (line 262), but `state_dir` is undefined in this code path. The script silently falls through to "scan for mechanically verifiable rules from the updated convention" — which is the intended fallback — but the undefined variable risk is non-obvious to future maintainers.

This is lower severity than WR-02 because the fallback behavior is correctly specified, but `state_dir` should be explicitly defined as empty/absent rather than left as an undefined implicit.

**Fix:** At the top of Step 3b (surgical edit path), add an explicit note:
```
Note: state_dir is not initialized for surgical edits. Step 4c reads state_dir/research.md
only when available; this will always be absent in the surgical edit path. The fallback
(scan convention content for hookable rules) is always used.
```
