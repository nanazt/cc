`# Spec Consolidator`

Brief introduction: You consolidate a single component's phase decisions into spec files. You receive structured dispatch from the `/consolidate` orchestrator.

`## Input Contract`

Document all XML dispatch tags the agent receives (the full table from the interfaces block above). For each tag, specify what the agent does with it:
- `<component>`: Used in output file paths, heading text, operation name prefixes
- `<sections>`: The explicit section list for context.md. Agent MUST produce exactly these sections in this order. Do NOT use internal defaults or add sections not in this list.
- `<conditional_sections>`: For each item, read phase documents and evaluate the natural-language `condition`. If condition is met, include the section. If not, exclude it. Log reasoning as HTML comment: `<!-- {section name}: {included|excluded} -- {reasoning} -->`
- `<rule_prefix>`: Use this prefix when promoting PR rules (e.g., if prefix is "CR", promote PR-1 to CR-1). This replaces the hardcoded "SR" from v1.

`## Output Format`

**context.md structure:**
```markdown
# {Component} Spec

Last consolidated: Phase {id} ({YYYY-MM-DD})

> Component rules ({prefix}-XX) defined in cases.md. Referenced, not duplicated.

## {Section Name from sections dispatch tag}

{Content matching the guide text scope}

{... for each section in the sections list ...}

{... conditional sections if included ...}
```

**cases.md structure:**

Last consolidated: Phase {id} ({YYYY-MM-DD})

## {Component}.{OperationName}

### Operation Rules

OR-1: {description} (Source: Phase {id})

### Side Effects

{side effects list}

### Cases

| Type | # | When | Then | Expected Outcome |
|------|---|------|------|-----------------|
| S | 1 | {condition} | {action} | {outcome} |
| F | 1 | {condition} | {action} | {outcome} |
| E | 1 | {condition} | {action} | {outcome} |
```

Use `{Component}.{OperationName}` format for all operation headings (per D-10 Rule 1).

`## Merge Rules`

Enumerate all 11 rules with v2 terminology. This section is critical -- the agent must follow these exactly:

**Rule 1 -- Operation-level replacement:** When a later phase re-specifies an operation (same `{Component}.{Op}` heading), the entire operation section replaces the prior version. No case-level merge.

**Rule 2 -- PR to CR promotion:** ALL Phase Rules (PR-N) from source CASES.md are mechanically promoted to Component Rules using the `<rule_prefix>` (e.g., CR-N). Renumber from highest existing CR + 1 to avoid collisions. (Changed from v1: SR -> CR per Phase 9 terminology.)

**Rule 3 -- TR exclusion:** Temp Rules (TR-N) are skipped entirely. Never enter specs/.

**Rule 4 -- R to OR transformation:** Operation Rules labeled R-N in source are renamed to OR-N in output. Consolidation-time transformation only.

**Rule 5 -- GR reference only:** Global Rules (GR-XX) in PROJECT.md are referenced with `See GR-XX`. Never copy content.

**Rule 6 -- Superseded operations:** For each entry in `<superseded_operations>`, remove the old operation section from existing spec. The Replacement column is for reference only -- consolidator's job is removal.

**Rule 7 -- Superseded rules:** For each entry in `<superseded_rules>`, skip the referenced PR during CR promotion.

**Rule 8 -- Section-level rewrite for context.md:** For each schema-defined section (from `<sections>`), if the new phase has content, rewrite the section. If no new content, leave existing section unchanged. (Changed from v1: "archetype sections" -> "schema-defined sections".)

**Rule 9 -- Provenance:** Every rule and decision entry includes `(Source: Phase {id})` or `(Source: Phase {id} D-{n})`.

**Rule 10 -- Forward Concerns exclusion:** Forward Concerns from CASES.md never enter specs/.

**Rule 11 -- Content exclusions:** Infrastructure setup, testing strategy, discussion rationale, research findings, planning artifacts.

`## Conditional Section Evaluation`

Per D-08: This agent evaluates conditional sections, not the orchestrator.

For each item in `<conditional_sections>`:
1. Read the phase documents listed in `<files_to_read>`
2. Evaluate the `condition` text against what you find (e.g., "component manages stateful entities with lifecycle transitions")
3. If the condition is met: include the section in context.md
4. If not met: exclude the section
5. Log your reasoning as an HTML comment at the end of context.md: `<!-- Conditional: {name}: {included|excluded} -- {brief reasoning} -->`

Superseded operations removed: {count} (list)
Superseded rules skipped: {count} (list)
Conditional sections: {included list} / excluded: {excluded list}
```

On failure:

`# E2E Flows Agent`

Brief introduction: You generate cross-component E2E flow documentation. You receive structured dispatch from the `/consolidate` orchestrator. This agent only runs when E2E flows are enabled in the project's schema.

`## Input Contract`

Document all XML dispatch tags:
- `<objective>`: Action description
- `<changed_components>`: JSON manifest: `[{"component": "auth", "operations_added": [...], "operations_removed": [...], "crs_promoted": [...]}]`. Note: uses "component" not "service", "crs_promoted" not "srs_promoted".
- `<spec_hashes>`: JSON from hash-sections.ts. Structure: `{ files: [{ path, sections: [{ heading, hash }] }] }`. The orchestrator computes these; this agent compares only, never computes hashes.
- `<existing_flows>`: JSON array of existing flow file paths (optional)
- `<new_flows>`: JSON array of developer-confirmed new flow names (optional)
- `<project_file>`: Path to PROJECT.md for component topology reference
- `<specs_dir>`: Path to specs/ directory root

`## Flow Format`

Each E2E flow file (`specs/e2e/{flow-name}.md`) contains these sections:

1. **Title and description** -- `# {Flow Name}` followed by one-sentence description
2. **Step Table** -- 6-column table: #, From, To, Action, Data, Ref
3. **Sequence Diagram** -- Mermaid sequenceDiagram matching the step table
4. **Error Paths** -- table: #, At Step, Condition, Response, Ref
5. **Spec References** -- table with Component, Section, Hash columns (note: "Component" not "Service" header)

Use "Component" column header (not "Service") in Spec References table.
File naming: hyphen-separated lowercase (`token-refresh.md`).
Granularity: Level B -- cross-component hops plus key internal logic with side effects.
Update strategy: full rewrite on update (not surgical edit).

`## Hash Comparison Logic`

For existing flows:
1. Read the Spec References table from the existing flow file
2. Compare each hash against the corresponding entry in `<spec_hashes>`
3. If ANY hash differs: the flow's dependencies changed, regenerate the flow
4. If all hashes match AND `<changed_components>` does not affect this flow's participants: skip the flow

`## Return Protocol`

On success:
```
## E2E FLOWS COMPLETE
Flows written: {count} ({list of file names})
Flows skipped (unchanged): {count} ({list})
New flows created: {count} ({list})
Stale flows updated: {count} ({list with changed dependency details})
```

On failure:
```
## E2E FLOWS FAILED
Reason: {what went wrong}
```

- [ ] Every flow has a Mermaid sequence diagram matching the Step Table

- [ ] Flow file names use hyphen-separated lowercase
- [ ] No flows reference operations that were superseded/removed
- [ ] New flows are only those listed in `<new_flows>` (developer-confirmed)
- [ ] All references use "component" terminology, not "service"
