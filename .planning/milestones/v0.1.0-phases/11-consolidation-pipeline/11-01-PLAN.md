**Frontmatter (per D-04, research Example 1):**
The YAML frontmatter must contain these exact fields:
- `name: consolidate` (changed from `consolidate-specs`)
- `description:` multi-line folded scalar (use `>`) describing per-component consolidation with triggers: after ship, spec consolidation, phase completed
- `argument-hint: "[phase-number]"`
- `allowed-tools:` list containing Agent, Bash, Read, Write, Glob, Grep, AskUserQuestion (7 tools total). Note: add Agent (new, for subagent dispatch), remove Edit (not needed)
- `disable-model-invocation: true`

**Body structure -- organize as follows:**

`<purpose>`: After a phase ships, consolidate its decisions into per-component spec files at `.planning/specs/{component}/`. Each spec file is the current truth about a component -- latest decision wins. Use "component" throughout, never "service" in user-facing text.

`<principles>`: Update from v1:
- Latest decision wins. New phase overrides prior decisions. Replace, don't append.
- Component-centric. Classify by component, not by phase.
- Provenance inline. `(Source: Phase {id})` on each entry.
- CR referenced, never copied. `See CR-01` only. (Changed from SR per Phase 9 terminology)
- Forward Concerns stay in phase CASES.md. Specs = confirmed decisions only.
- Use `{Component}.{OperationName}` format. Required for briefer grep discoverability. (Per D-10 Rule 1)

`<what_to_exclude>`: Same as v1 -- infrastructure, testing strategy, discussion rationale, research findings, planning artifacts.

`<procedure>`: The 7+2 step pipeline. Each step is an H2 with specific instructions:

**Step 1: Read Phase Documents, Parse Schema, Discover Components**
Per D-05: Call schema-parser.ts via Bash:
```
deno run --allow-read tools/schema-parser.ts .planning/consolidation.schema.md
```
If schema file missing: offer bootstrap via `deno run --allow-read --allow-write tools/schema-bootstrap.ts .planning/consolidation.schema.md`. On bootstrap, ask developer to populate components, then re-parse.
Parse JSON stdout. Extract:
- `meta.e2eFlows` (boolean) -- store for Step 3.5/4 gating
- `meta.rulePrefix` (string, e.g. "CR") -- use for rule promotion prefix
- `components[]` -- component registry
- `sections` -- section structure map

Resolve section structure per component:
- If `component.type` is non-empty -> use `sections[component.type]`
- If `component.type` is empty -> use `sections["default"]`

Component discovery per D-06 (no keyword fallback):
1. Scan CASES.md (or CASE-SCRATCH.md) for `## {Component}.{Op}` heading patterns. Extract unique component names.
2. If zero: scan CONTEXT.md for component names matching schema `components[].name`.
3. If still zero: AskUserQuestion -- "Could not identify affected components. Which components does this phase affect?"

Cross-reference discovered components against schema:
- In schema -> proceed
- Not in schema -> per D-13: "Component '{name}' discovered in phase documents but not in schema. Add to schema?" (AskUserQuestion). If yes, add row to schema Components table.

Read existing spec files if they exist: `specs/{component}/context.md`, `specs/{component}/cases.md`.

Out-of-order consolidation check: compare existing spec's `Last consolidated: Phase {id}` vs source phase. Older source = warn, don't block.

**Step 2: Dispatch Per-Component Spec-Consolidator Agents (Parallel)**
Per D-07/D-08/D-09: Build dispatch prompt for each component with these XML tags:

| Tag | Required | Contents |
|-----|----------|----------|
| `<objective>` | Yes | "Consolidate Phase {id} decisions for {component} into specs/{component}/." |
| `<component>` | Yes | Component name (lowercase) |
| `<sections>` | Yes | JSON array of context sections from schema resolution (the `context[]` array from the resolved SectionBlock) |
| `<conditional_sections>` | Yes | JSON array of conditional sections with condition text (the `conditional[]` array). Even if empty, include as `[]`. |
| `<rule_prefix>` | Yes | The `meta.rulePrefix` value (e.g. "CR") |
| `<files_to_read>` | Yes | Phase CONTEXT.md, CASES.md, PROJECT.md paths |
| `<existing_spec>` | No | Path to existing specs/{component}/context.md (omit for new) |
| `<existing_cases>` | No | Path to existing specs/{component}/cases.md (omit for new) |
| `<output_context>` | Yes | `.planning/specs/{component}/context.md` |
| `<output_cases>` | Yes | `.planning/specs/{component}/cases.md` |
| `<phase_id>` | Yes | Phase identifier string |
| `<superseded_operations>` | No | JSON from CASES.md Superseded Operations table |
| `<superseded_rules>` | No | JSON from CASES.md Superseded Rules table |

Extract superseded_operations and superseded_rules from source CASES.md if present.
Dispatch all agents in parallel using the Agent tool.
Collect return messages. Parse `## CONSOLIDATION COMPLETE` or `## CONSOLIDATION FAILED`.

Error handling per D-20: fail-fast + selective retry.
- If ANY agent fails: halt, report, offer retry (failed agent only) or abort.
- Abort: `git checkout -- .planning/specs/`

**Step 3: Collect Results, Update INDEX.md**
Parse each consolidator return: component name, files written, operation count, promoted CRs (per D-10 Rule 2: PR->CR not PR->SR), supersessions.
Build `<changed_components>` manifest JSON.
Write `.planning/specs/INDEX.md` using v2 format per D-11:

```markdown
# Spec Index

Last updated: {YYYY-MM-DD}

## Components

| Component | Type | Description | Files | Last Consolidated |
|-----------|------|-------------|-------|-------------------|
| {name} | {type or empty} | {description} | [context]({name}/context.md) [cases]({name}/cases.md) | Phase {id} ({date}) |

## E2E Flows

| Flow | File | Participants | Last Updated |
|------|------|-------------|--------------|
{rows if any, or "No E2E flows."}

## Operation Index

| Operation | Component | Cases Source | Phase |
|-----------|-----------|-------------|-------|
| {Component}.{Op} | {component} | [cases.md]({component}/cases.md) | {id} |
```

Type column always displayed per D-11 (even if empty for untyped components).
INDEX.md is fully rewritten each run, not surgically edited.

**Step 3.5: Identify E2E Flows (Conditional)**
Per D-04 and research Pattern 3: check `meta.e2eFlows` from Step 1.
- If false: skip Steps 3.5 and 4 entirely. Log: "E2E flows disabled in schema. Skipping."
- If true: proceed with flow discovery.

When enabled:
1. Scan existing `specs/e2e/` for current flow files.
2. For changed components, check Dependencies section for cross-component references.
3. New flow candidates -> AskUserQuestion for confirmation.
4. Build `<existing_flows>` and `<new_flows>` lists.

**Step 3.7: Orphan Directory Cleanup**
Per D-14/D-15:
1. Scan `specs/` for component directories.
2. For each, count operations in the component's spec files.
3. If 0 operations (all superseded/moved): flag as orphan candidate.
4. Present orphans to developer: AskUserQuestion "Orphan spec directories (0 operations): {list}. Remove?"
5. If confirmed, delete directory.
6. Per D-15: also offer to remove from schema file.

**Step 4: Dispatch E2E Flows Agent (Conditional)**
Only runs if `meta.e2eFlows` is true (checked in Step 3.5).

1. Verify Deno: `which deno`. Missing -> report error, skip E2E, do NOT rollback specs.
2. Compute hashes: `deno run --no-lock --allow-read tools/hash-sections.ts {all spec files}`. Pass ALL spec files.
3. Parse hash JSON output.
4. Build dispatch prompt with XML tags for e2e-flows agent (see agent contract).
5. Dispatch e2e-flows agent (sequential, after Step 3).
6. Parse return.

Error handling: E2E failure does NOT rollback service spec consolidation. Mark "E2E flows: SKIPPED" in summary.

**Step 5: Dispatch Spec-Verifier Agent (Conditional)**
Per D-21: Check if `agents/spec-verifier.md` exists (use Glob or file read).
- Absent: log "Verification skipped (spec-verifier agent not yet created)". Mark "UNVERIFIED".
- Present: dispatch spec-verifier agent with full XML contract.

Include a note in the SKILL.md: `<!-- Remove this skip branch once agents/spec-verifier.md exists in the agents/ directory -->`

Error handling: verifier failure -> UNVERIFIED marking, proceed. Read-only agent, no rollback.

  - {list or "None."}

Present via AskUserQuestion: "Phase {id} consolidation ready. {N} components, {N} operations, {N} findings. Confirm commit?"

T1 findings warning if present.
