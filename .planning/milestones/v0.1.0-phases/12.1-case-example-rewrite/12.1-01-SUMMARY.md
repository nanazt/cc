# Phase 12.1 Plan 01: SKILL.md and step-discuss.md Neutralization Summary

**One-liner:** Replaced all technology-biased examples in SKILL.md and step-discuss.md with structural placeholders — global adaptation instruction, open side effect format, structural error naming rule, and neutralized canonical flow diagram.

## What Was Done

### Task 1: SKILL.md — 7 changes applied

1. **Global adaptation instruction (D-19)** added to `<philosophy>`: authoritative statement that all examples use structural placeholders and Claude must adapt content to the host project's actual interfaces, protocols, and terminology.

2. **Scope guardrail examples (D-14):**
   - `"What if the token is expired?"` → `"What if the [credential/precondition] is [invalid state]?"`
   - `"Should we use a middleware for this?"` → `"Should we use [implementation approach] for this?"`

3. **Developer-proposed design intent example (D-14):**
   - `"use a parameterized function for two middleware variants"` → `"[implementation approach for a specific design decision]"`

4. **AskUserQuestion format example (D-14):**
   - `"Logout cases review. Missing anything?"` → `"[OperationName] cases review. Missing anything?"`

5. **Canonical example Interface field (D-04, D-12):**
   - `Auth: any authenticated role (standard middleware)` → `Caller: [who/what can invoke this]`

6. **Canonical flow diagram failure branches (D-04):**
   - Three `→ 400 bad request / 401 unauthorized / 500 internal error` lines → `→ [error outcome (ErrorName)]`

### Task 2: step-discuss.md — 7 changes applied

1. **TR classification example (D-14):**
   - `"use mock SMTP during development"` → `"[implementation approach that is explicitly temporary]"`

2. **Inherited concerns example (D-14):**
   - `RefreshToken R4` → `[OperationName rule-ref]`

3. **6 fixed side effect categories replaced with open format (D-08):** Removed Domain events, Related entity updates, Cache mutations, Audit/logging, Notifications, External system calls. 4. **Stale "omit categories" guidance (D-14):** Removed the `Notifications`/`internal data migration operation` example — replaced with the SE_ working-label guidance which was already partly present.

5. **Multi-protocol error example block replaced with structural rule (D-10, D-11):** Entire block (HTTP/gRPC/domain/opaque examples) replaced with: structural rule `[error description] (ErrorName)`, domain-level explanation, and specificity criterion "be specific enough that a test can assert against it."

6. **Forward column example (D-14):**
   - `->3B:AddPasskey` → `->XX:[OperationName]`

7. **CASE-SCRATCH.md Side Effects template (open question 1 from RESEARCH):**
