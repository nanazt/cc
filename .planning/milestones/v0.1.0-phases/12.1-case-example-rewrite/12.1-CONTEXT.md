<decisions>
## Implementation Decisions

### Unified principle
- **D-01:** Teach STRUCTURE and SPECIFICITY LEVEL only. Never prescribe domain-specific content. All examples, templates, and guides must instruct Claude on the format and quality bar, then direct it to adapt content to the host project's actual interfaces, protocols, and terminology.
- **D-02:** "Be specific enough that a test can assert against it" is the specificity criterion for all Expected Outcomes (success and failure).
- **D-03:** No distinction between "educational examples" and "template examples" — all examples follow the same abstraction principle. Claude already knows about HTTP, gRPC, CLI patterns from training data; the skill's job is to direct behavior, not teach domain knowledge.

### Template examples (canonical flow, output format)
- **D-04:** SKILL.md canonical flow diagram: replace HTTP status codes (400, 401, 500) with structural placeholders (`[error outcome (ErrorName)]`). Replace `Auth: any authenticated role (standard middleware)` with `Caller: [who/what can invoke this]`.
- **D-05:** step-finalize.md output_format: Expected Outcome examples replaced with structural rule + specificity guide. No concrete domain/protocol examples in the template cells.
- **D-06:** Error consistency table (step-finalize.md 4a): replace HTTP status code cell values with checkmarks or error names. The table compares error handling consistency across operations — the comparison is valid regardless of protocol.
- **D-07:** Success Expected Outcome guidance: describe "result + state change + side effects" at a level tests can assert. Not just "Success" — all observable effects must be specified. Replace `"Success; 'entity.created' event emitted"` with structural guidance.

### Side effects section
- **D-08:** Replace ALL 6 fixed side effect categories with open format. The current categories (Domain events, Related entity updates, Cache mutations, Audit/logging, Notifications, External system calls) anchor Claude to specific architecture patterns. Use an open "list expected side effects for this operation" approach and let Claude adapt categories to the host project's actual architecture.

### Error naming convention
- **D-09:** Domain-level is the canonical form for Expected Outcomes. Protocol mapping (HTTP status codes, gRPC error codes, CLI exit codes) is the implementation's concern, not the behavioral specification's.
- **D-10:** The structural rule is: `"[error description] (ErrorName)"`. Omit parenthetical only when intentionally opaque. This rule is taught without concrete domain examples.
- **D-11:** step-discuss.md's multi-protocol error example block (lines 271-275, currently showing HTTP/gRPC/domain/opaque) is replaced ENTIRELY with the structural rule from D-10. No per-line neutrality exceptions — even the neutral examples (line 272 "file too large", line 275 "credentials invalid") are removed because the block itself carries a "protocol showcase" framing. The structural rule + specificity criterion (D-02) is sufficient for Claude to produce correct error outcomes.

### Interface section field
- **D-12:** `Auth:` field renamed to `Caller:` in the canonical example and all guide text. "Caller" is universal — HTTP request sender, CLI invoker, library consumer, process caller all qualify. Access control details are expressed in the field value, not the field name.
- **D-13:** `Auth Required:` field in the output format (step-finalize.md Operation header) changed to `Caller:` with structural placeholder value.

### Inline e.g. examples
- **D-14:** All technology-biased inline `(e.g., ...)` examples across all 5 files are replaced with structural placeholders. Not replaced with "neutral-sounding concrete examples" (which still anchor Claude to specific domains), but with structural expressions like `[OperationName]`, `[error description] (ErrorName)`, `[implementation approach]`.
- **D-15:** Non-biased inline examples are left as-is. Bias is defined as: assumes a specific communication protocol, storage mechanism, auth mechanism, architecture pattern, or project type.

### Validator classification criteria
- **D-16:** Language Pattern Signals category structure (Behavioral signals, Architectural constraint signals) is kept intact. Only the inline examples within each category are neutralized. The classification logic itself is sound and technology-neutral — it's the illustrative examples that carry bias.
- **D-17:** `"Explicit status codes"` category name is retained (status codes exist across protocols). Example changes: `"returns 401"` → `"returns [status/error code]"`, `"all endpoints"` → `"all operations"`.

### Probing language
- **D-18:** ZOMBIES probing language in step-discuss.md is NOT in scope. "Zero → One → Many" progression and its specific questions ("What about bulk scenarios?", "What does a list with many results look like?") are testing technique language, not architecture assumptions. Already universal.

### Adaptation instruction strategy
- **D-19:** Hybrid approach — global instruction in SKILL.md + selective inline annotations at non-obvious sites.
  - **Global:** One authoritative statement in SKILL.md (before or inside `<philosophy>`) that all examples use structural placeholders and Claude must adapt content to the host project's actual interfaces, protocols, and terminology.
  - **Inline:** Per-site adaptation annotations ONLY where the gap between placeholder and correct output is non-obvious — specifically the error naming convention (step-discuss.md) and Expected Outcome guidance (step-finalize.md). Mechanical placeholders like `[OperationName]` and `[Component]` rely on the global instruction alone.

### Phase scope
- **D-20:** 5 files in scope for content rewrites (see deliverables above). All changes are content rewrites within existing file structures — no structural/architectural changes to the skill. README.md is deleted entirely.
- **D-21:** README.md deleted in Phase 12.1. Full rewrite deferred to a future phase. README is user-facing documentation, not a system prompt consumed by Claude — no functional impact on /case behavior.

<specifics>
## Specific Ideas

- The unified principle emerged during discussion: initial approach was "rotate diverse domain examples" but this was challenged as still anchoring Claude to specific domains. Final principle: teach structure, not content.
- step-discuss.md's existing multi-protocol error block (lines 271-275) was initially considered for "educational exemption" but this was rejected — no special treatment for any examples, even neutral ones within a biased framing.
- "Be specific enough that a test can assert against it" was identified as the key specificity criterion that replaces concrete examples.
- Side effects section: all 6 fixed categories replaced with open format. The "omit what doesn't apply" approach was rejected as insufficient — fixed categories still anchor Claude to specific architecture patterns.
- README.md deletion chosen over in-place neutralization because README is user docs (not Claude system prompt), and partial fixes would leave incomplete documentation. Clean deletion + future rewrite is simpler.
- Adaptation instruction uses hybrid strategy because Claude's attention decays across multi-file skill sessions — global instruction alone may be insufficient for non-obvious adaptation sites.
