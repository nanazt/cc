## Format

Use Conventional Commits 1.0.0 for every commit message:

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

Types: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `perf`, `ci`, `build`, `style`

## Scope

The scope in parentheses must name a real component, module, or directory in the codebase.

Bad: `feat(PROJ-123): ...`, `fix(sprint-4): ...`, `chore(workflow): ...`
Good: `feat(auth): ...`, `fix(parser): ...`, `chore(deps): ...`

## Subject Line

- Start with lowercase after the colon: `fix(api): handle null response` not `fix(api): Handle null response`
- Use imperative mood: "add feature" not "added feature" or "adds feature"
- Write it to be understood without access to any internal planning documents — describe the change, not the planning activity that produced it

Bad: `docs(spec): complete plan 02 orchestrator work`
Good: `docs(spec): add orchestrator retry logic to spec`

## Body

When a body is included:

- Explain why the change was necessary, not what the diff shows
- Hard-wrap at 72 characters per line
- Separate from the subject with a blank line

Bad body: "Changed the retry count from 3 to 5 and added a backoff multiplier."
Good body: "Three retries proved insufficient under network degradation. Five retries with backoff brings the failure rate within acceptable bounds."

## No Project Management References

Never include internal planning references in commit messages — not in subject, not in body, not in footers. This applies to:

- Phase numbers: `phase 11`, `phase 9`
- Plan IDs: `11-01`, `plan 02`
- Requirement IDs: `PROJ-01`, `MODEL-03`
- Decision IDs: `D-03`, `D-15`
- Progress fractions: `plan 1/3`, `progress 71%`
- Sprint numbers, ticket IDs from any workflow system

Describe what was done in plain language. If a workflow tool suggests a message containing these patterns, rewrite it before committing.

## Breaking Changes

Signal breaking changes with `!` after type/scope or with a `BREAKING CHANGE:` footer:

```
feat(api)!: remove deprecated endpoint

BREAKING CHANGE: /v1/users endpoint removed. Use /v2/users instead.
```

Include migration instructions when the breaking change requires caller updates.

## Issue References

Put issue references in the footer, not the subject line:

```
fix(parser): handle empty input gracefully

Fixes #42
```

## Atomic Commits

Each commit represents one logical change. Do not mix unrelated changes in a single commit. Each commit must leave the codebase in a working state.
